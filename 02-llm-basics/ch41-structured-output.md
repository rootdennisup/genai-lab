# 结构化输出

## 1 基本概念
结构化输出（Structured Outputs）是指让模型返回遵循特定格式（通常是 JSON）的数据，而非纯文本回复。这是实现 AI 与后端程序自动对接的基石。

- **API 关键参数** (``response_format``)：在调用 Chat Completions API 时，通过设置 ``response_format`` 参数可以开启此功能，确保模型生成的 JSON 数据符合预定义的约束。
- **JSON 模式** (JSON Mode)：通过将 ``response_format`` 设置为 ``{"type": "json_object"}``，可以确保模型生成的输出是有效的 JSON 字符串，防止因夹杂自然语言描述而导致解析失败。
- **输出合同** (Output Contracts)：针对 GPT-5 系列模型，定义明确的“输出合同”至关重要，它规定了模型必须交付的数据形态和字段质量。

## 2 Schema 约束：定义数据的“形状”
模型通过 JSON Schema 格式来约束模型的响应。

- **函数调用**(``Function Calling``)：这是获取结构化数据最可靠的机制。开发者定义函数的 ``parameters`` 对象（包含 type、properties 和 required 必填项），模型则负责填充这些参数。
- **字段定义细节**
  - **类型描述**：为每个字段提供清晰的 description（如“这是一个 SQL 查询”），能显著提高生成准确率。
  - **必填 vs 可选**：在 Schema 中明确划分 required 字段，有助于模型在信息不足时做出决策。
  - **提取完备性**：在处理长文档（如 PDF）提取时，应显式要求模型保持“提取完备性”，防止其在生成过程中由于 Token 限制而中途“偷懒”。

## 3 提示词技巧：确保“纯净”输出
即使开启了 JSON 模式，优秀的提示词仍是成功的保障：

- **负面提示** (Negative Prompts)：明确告知模型“不要在 JSON 代码块之前或之后添加任何解释文字”或“除了 JSON 以外不要回答任何其他内容”。
- **特定后缀引导**：要求模型的输出必须能被 json.loads 接受，或者将重复的格式指令放在提示词的末尾（重复指引法）。
- **结果导向设计**：定义“什么是好的输出”，让模型自主规划填充 JSON 字段的推理路径，而非手把手教它提取步骤。

## 4 输出校验与验证循环
在大规模生产环境中，不能完全信任单次生成的输出。

- **自检指令**：在提示词中增加一个**自检步骤**（例如：“在返回 JSON 前，检查是否缺失了必填的 ID 字段”），利用模型的推理能力进行格式纠偏。

- **验证循环机制**：
  - **得出初步结果**：模型生成 JSON。
  - **轻量级验证**：在交付给最终业务逻辑前，指令模型或代码层针对**高风险字段**（如财务金额、SQL 语法）进行核查。
  - **发现冲突并修正**：如果发现违反约束，模型应重新规划路径并修正输出。

- **失败处理与稳定性**：
  - **重试机制** (Retry)：针对格式错误的“软失败”，应配合 **指数退避**（Exponential Backoff） 算法进行逻辑重试，给服务器和模型恢复的时间。
  - **锁定模型快照**：生产环境建议锁定特定的模型快照版本（如 gpt-4-0613），防止模型更新导致的 **JSON 字段漂移或解析逻辑中断**。
  - **Token 预检**：使用 tiktoken 库预先计算输入规模，确保返回的 JSON 数据不会因为超出上下文窗口而被截断，导致 JSON 结构不完整。
