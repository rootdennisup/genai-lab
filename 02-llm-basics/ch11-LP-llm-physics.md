Token 与分词机制 是大模型应用开发的最底层基石，它不仅决定了模型如何“阅读”和“思考”，还直接关系到你的开发成本、响应速度和模型性能。

## 1 Token（标记）

### 1.1 基本概念
- ``Token`` 是大模型处理文本的原子单位。而在模型眼中，它看到的不是字符串，而是一个个**标记序列**。

- **Token 的表现形式**：Token 并不等同于单词。它可能是完整的单词、单词的一部分（子词/Subword）、空格或标点符号。例如，“weather”可能会被拆分为 wea 和 ther。

- **直观换算经验**：
  - **英文**：100 个 Token 约等于 75 个英文单词。
  - **字节**：在某些场景下，可以通过 **字节数 / 4** 来粗略估算 Token 数。

### 1.2 Token 计费机制
与传统云服务按调用次数计费不同，LLM API 严格执行按量计费，而这个“量”就是 Token 数量。

- **双向汇总计费**：单次 API 调用的总成本 = 输入文本 (Prompt) 的 Token 数 + 输出文本 (Completion) 的 Token 数。

- **计费透明性**：API 返回的响应对象中包含 ``usage`` 字段，明确列出 ``prompt_tokens``（输入数）、``completion_tokens``（输出数）和 ``total_tokens``（总数）。

- **参数影响**：如果设置 n > 1（生成多个候选回答），输入 Token 数不变，但输出 Token 数会随生成的回复数量成倍增加。

- **价格差异**：不同模型的单价不同。例如，GPT-5 的单价通常远高于 GPT-4。开发者通常需要根据每 1000 个 Token 的单价来预估业务成本。


## 2 分词器（Tokenizer）
- 分词器（Tokenizer）是模型处理文本的第一道工序，也是开发者理解模型行为和成本控制的核心。
- 由于计算机无法直接理解字符，必须通过分词器将文本转换为模型能处理的**数字序列**。

### 2.1 基本概念
- **标记化**（Tokenization）：将原始文本（如句子、段落）切分成一个一个名为 Token（标记） 的单元的过程。

- **语义独立性**：分词器的目标是**保证每个标记拥有相对完整和独立的语义**，以便后续生成**嵌入**（Embeddings）。

- **分词器差异**：几乎每个语言模型都配有自己特定的分词器。例如，GPT-4 和 GPT-5 的分词方式可能不同。

### 2.2 处理流程
当模型接收到一段提示词（Prompt）时，分词器的处理流程如下：

- **原始文本**（输入） →

- **分词器处理**（标记化）：
  - **拆分 (Split)**：按照特定算法将原始文本切分为 Token 序列。 
  - **映射 (Map)**：将切分出的 Token 转换为**模型内部字典**中对应的**唯一数字 ID**。

- **后续环节**：数字 ID 随后被转换为**数值向量**（Embeddings），供模型进行语义关系处理和下一个 Token 的预测。

### 2.3 分词器与后续环节的关系
后续环节主要有两种：

- **文本补全**：其本质是模型在预测时，是根据已有的 Token 上下文，为字典中所有潜在的后续 Token **分配概率**，并逐个选择概率最高的 Token 形成完整响应。
- **衔接嵌入（Embeddings）**：由于模型只能处理数值，分词器切分出的 Token 随后会被转换为数值向量（即嵌入），从而让计算机能够高效处理它们之间的语义关系。

### 2.4 模型内部字典
模型内部字典是一个**静态的映射表**，它记录了分词器能够识别的所有基本单位（```Tokens```）与唯一**数字 ID** 之间的对应关系。

- **唯一性**：字典中的每一个 Token（如一个单词、子词或标点）都对应一个唯一的整数 ID。
- **桥梁作用**：由于模型底层的 ```Transformer``` 架构只能处理**数值向量**（Embeddings），分词器的任务就是查表，将文本序列转换为数字 ID 序列，再由模型进一步处理。
- **构成要素**：字典不仅包含常见的字词，还包含空格、标点符号以及特殊的占位符（如结束符 ``<|endoftext|>``）。

#### 不同模型的内部字典不相同

- **模型特异性**：分词算法（如 ``BPE、WordPiece``）和训练数据的不同，导致了词表的多样性。

- 几乎每个语言模型都配有自己**特定的分词器**（Tokenizer），例如：如阿里的 Qwen 与 OpenAI 的 GPT。

- 为了提升效率（用更少的 Token 表示更多的内容）或增强对特定语料（如数学、代码）的处理能力，模型厂商通常会升级分词器并更新内部字典。例如：GPT-4 和 GPT-5 的分词方式可能不同。

#### 理解字典的差异具有重要的工程意义
- **Token 换算率不同**：不同模型的字典效率不同。虽然英文有一个通用的经验法则（100 个 Token ≈ 75 个单词），但具体比例会随模型词表的设计而波动。

- **成本与窗口计算**：API 计费和上下文窗口限制都是基于 Token 数量的。由于字典不同，同一段文本在 GPT-4 中可能是 10 个 Token，但在另一个模型中可能是 15 个，这直接影响成本统计。

- **工具选择**：开发者在本地进行 Token 预计算时，必须使用与目标模型匹配的库（如 OpenAI 提供的 tiktoken 库专门用于其特定模型）。


## 3 Token 成本统计与控制
在生产环境中，Token 数量直接决定了应用的可用性和经济性。

- **上下文窗口限制**：模型能处理的“内存”是有限的。每个模型都有其上下文窗口上限（如 8k、128k 或 1M 标记）。输入和输出的总和必须在这个上限之内，否则模型将报错或丢失最早的上下文。

- **延迟监控**：Token 数量越多，模型预测所需的迭代次数越多，导致响应时间（延迟）正向增加。

- **成本控制**：可以通过在 API 调用中设置 ``max_tokens`` 参数来限制模型生成的最大长度，防止因模型过度发挥而产生高额账单

- **核心工具：tiktoken**：
    - **本地预检**：Tiktoken 允许你在正式发起网络请求前，在本地计算字符串的 Token 数。
    - **作用**：帮助进行精确的成本估算、预防超出模型窗口导致的失败，以及在上下文过长时进行精确截断。


## 4 上下文窗口（Context Window）

### 4.1 什么是上下文窗口
上下文窗口是大语言模型在生成新信息时可以处理的 **“内存”限制**（物理限制）。它以 Token 为单位，规定了模型在单次预测任务中能够“同时看到”的最大文本范围。

- **物理隐喻**：可以把它想象成模型的“短期记忆”。只有在这个窗口内的信息，模型才能理解其中的语义关系并保持逻辑连贯。

- **无状态特性**：需要注意的是，OpenAI 的 API 是**无状态**的，它不会记住之前的请求。因此，如果想进行多轮对话，必须每次都将之前的对话历史完整地作为上下文发送给模型。

- **不同模型的窗口容量可能存在差异**，这直接决定了它们处理长文档或长对话的能力，例如：
  - GPT-4 系列：提供 8,192 和 32,768（约 2.4 万个单词或 40 页文本）两种版本。
  - GPT-4.1 ：窗口大小已显著提升至 12.8 万（128k） 甚至 100 万（1M） 个标记。

### 4.2 长上下文下的“Lost in the middle”现象与推理性能波动
随着上下文窗口从早期的几千 Token 演进到现在的 1M（100万）甚至更高，开发者面临的核心挑战已不再仅仅是“装不装得下”，而是模型在**处理海量信息时的召回质量与推理稳定性**。

#### “Lost in the middle”（迷失中段）现象
指的是大模型在处理长文本时，对输入内容开头（Beginning）和结尾（End）的信息识别与召回效果最好，而对于**位于文本中间的信息，其关注度和处理能力会显著下降**。

- **物理机制**：虽然模型物理上可以“看到”整个窗口内的所有 Token，但**注意力机制**（Attention Mechanism）在长序列中会将**权重分散**。当上下文过于冗长且密集时，模型容易在“信息流”中迷失，导致关键事实的检索失败。
- **性能表现**：在实际测试中（如 Needle In A Haystack/大海捞针测试），如果目标信息被放置在提示词的中段，模型的回答准确率往往会低于将其放在开头或结尾的情况。
- **后果**：直接导致模型在处理长文档分析、多文档评估或跨段落的复杂任务时，出现漏掉关键细节、产生幻觉或给出逻辑不连贯答案的问题。

#### 推理性能波动（Performance Fluctuation）
长上下文不仅影响记忆，还会直接导致模型的推理能力产生剧烈波动，主要体现在以下几个维度：

- **检索与多跳推理退化**：模型在长上下文下的性能在需要 **精确检索（Precise Retrieval）或多步推理（Multi-hop Reasoning）**的任务中会显著下降。例如，执行复杂的图搜索或跨多处信息的逻辑推导时，上下文越长，出错概率越高。
- **“Lost in the scroll”错误**：在密集的上下文环境下，模型可能会因为信息过载而无法维持长轨迹任务的稳定性，导致推理路径偏离最初的目标。
- **感官延迟增加**：Token 数量越多，模型预测所需的迭代次数和计算开销就越大，这会导致响应时间显著变长，感官延迟（Perceived Latency）增加，甚至可能触发 API 超时。

#### 针对性的工程对策
为了缓解上述性能衰减，开发者通常需要采取以下上下文工程策略：

- **指令位置优化**：在长文本场景下，理想的策略是**将核心指令同时放置在上下文的开头和结尾**。如果只能放一次，放在开头的效果通常优于放在结尾。
- **上下文压缩（Compaction）**：这是针对 GPT-5 系列模型的一项核心技术。通过“损失感知（Loss-aware）”的压缩方式，将冗长的历史记录转化为**加密且精简**的 ``encrypted_content``。这不仅能节省 Token 消耗，还能维持推理的稳定性，减少模型在长对话中的遗忘。
- **强制摘要与重锚定**：对于极长任务，指令模型在中间步骤先生成摘要或重新梳理已知事实（Re-grounding），可以有效减少“信息迷失”导致的错误。
- **RAG 与分块策略**：与其将整本书塞进窗口，不如利用**向量数据库（RAG 架构）**检索出最相关的“精华”片段喂给模型，从而人为缩短上下文长度，保证模型专注于最关键的信息。
- **调节推理努力**（Reasoning Effort）：对于需要深度思考的长轨迹任务，可以适当提高 ``reasoning_effort`` 参数（如设为 high 或 xhigh），赋予模型更多的“思考 Token”来应对复杂推理，提升自主解决问题的毅力。
