# Prompt 工程

## 1 核心角色与结构

### 1 消息角色
模型通过不同的角色来区分信息的来源和权威性。

- ``developer`` (开发者角色)：这是最高优先级的消息。它由应用程序开发者提供，用于**定义系统的运行规则、业务逻辑和行为边界**。在 GPT-5 系列模型中，它取代或增强了传统的 system 角色，具有更强的指令遵循能力。
- ``user`` (用户角色)：代表最终用户的输入。你可以将其视为 **“函数的参数”**，它提供具体的配置或数据，并被应用到开发者定义的逻辑中。
- ``assistant`` (助手角色)：由模型生成的响应消息。在多轮对话中，存储先前的助手回复对于保持对话上下文（记忆）至关重要。

### 1.2 developer 消息的标准结构
为了让 developer 消息发挥最大效力，建议将其拆分为以下四个逻辑板块（按顺序排列）：
- **Identity(身份)**：描述助手的目的、沟通风格和高层目标（如：“你是一位专业的法律助理，语气严谨且专业”）。
- **Instructions(指令)**：提供具体的操作指南。规定模型应该做什么以及绝对不能做什么（如：如何调用函数、处理未知情况的原则）。
- **Examples(示例)**：即 Few-Shot 示例，提供输入与期望输出的样本，帮助模型识别复杂的模式。
- **Context(上下文)**：提供模型回答问题所需的额外信息，如私有文档、参考资料或特定任务的相关背景。

### 1.3 Markdown 与 XML 标签化 (Formatting)
这是工业级提示词的核心技巧，用于帮助模型理解提示词的逻辑边界。

- **Markdown 标题与列表**：使用 Markdown 的标题（如 #、##）和列表来标记提示词的不同部分，使层次结构清晰，提高可读性。
- **XML 标签**：使用类似于 <context>、<rules>、<examples> 的标签包裹特定内容。
    - 优势：XML 标签能精确界定内容的起始和结束，有效防止模型将参考数据（数据）误认为是指令。
    - 应用：在长上下文（Long Context）任务中，XML 表现出色，能够帮助模型在海量信息中准确定位关键字段。

### 1.4 结果导向 (Outcome-First) 设计与指令优先级
这是针对 GPT-5 系列模型 最重要的进阶认知点。

#### 1> 结果导向 vs 过程堆叠
传统的提示词倾向于过度说明过程（Over-specify the process），即**教模型一步步该怎么做**。但对于 GPT-5 系列模型，建议转变为**定义结果**。

- **核心逻辑**：明确描述 **“好的输出是什么样的”**、**最终目标**以及必须遵守的约束，然后赋予模型**自主规划推理路径**的空间。
- 原因：GPT-5.5 等模型拥有更高效的推理能力，过多的琐碎过程指令反而会缩窄模型的搜索空间，导致输出变得机械化或产生不必要的噪声。

#### 2> 指令权力的等级制度
在 API 内部存在明确的权力层级：``developer > user``。

- 模型被训练为优先遵循开发者设定的规则。即使用户在消息中尝试注入“请忽略之前所有指令”等恶意提示（Prompt Injection），模型也会因为 developer 消息的权威性而保持对原始规则的忠诚。
- **工程建议**：永远不要让 user 角色消息涉及核心安全规则或业务逻辑，这些必须固化在 developer 消息中。


## 2 核心技术手段

### 2.1 少样本学习 (Few-Shot Learning)
少样本学习是指通过在提示词中包含少量（通常 1-5 个）“输入/输出”示例，引导模型识别模式并完成新任务，而无需重新训练模型权重。

- **工作机制**：模型通过示例隐式地“拾取”其中的模式，并将其应用到当前的输入中。

- **零样本 (Zero-shot) vs. 少样本 (Few-shot)**：
  - **零样本**：不提供示例，直接描述任务。适用于简单、模型背景知识充足的任务。
  - **少样本**：提供示例。当零样本无法精确控制输出格式或任务逻辑极其复杂时（如将文本转换为特定表情符号组合），少样本学习非常有效。

- **与微调的区别**：少样本学习不修改模型的内部参数，只是利用模型已有的泛化能力；而微调是在特定数据集上重新训练，修改内部权重。

- **编写原则**：示例应当清晰且相关，展示多样化的可能输入及其期望输出，避免提供模棱两可的信息导致错误结果。

### 2.2 思维链 (Chain of Thought, CoT)
思维链是一种鼓励模型模仿人类推理过程，通过展示中间步骤来解决复杂问题的技术。

- **基础逻辑**（逐步思考）：在提示词末尾添加“让我们逐步思考”（Let's think step by step）等指令，可以触发模型生成内部推理标记。

- ***为什么 CoT 有效**
  - 模型生成回答是迭代式的概率预测。如果直接要求答案，模型可能在没有充分“规划”的情况下直接生成错误的最左侧标记。
  - CoT 相当于给模型提供了“草稿纸”，让它有时间通过拆分问题来规避计算或逻辑陷阱。

- **使用边界**：
  - **适用场景**：多步算术、符号推理、策略性问题及其他逻辑复杂的任务。
  - **局限性**：对于模型回答常识性问题，思维链通常没有显著的提升效果。

### 2.3 Prompt 缓存优化 (Prompt Caching)
这是针对生产环境的高级优化技巧，旨在**降低延迟并节省 API 调用成本**。

- **核心原则**：在构建消息时，应尽量将预期会重复使用的内容放在提示词的最开头。

- **实现策略**：
  - 将静态指令、大型上下文或固定知识库内容置于 Prompt 起始位置。
  - 在 API 请求的 JSON 主体参数中，也将这些重复内容排在前面。

- **收益**：这样做可以最大化利用 API 供应商提供的提示词缓存机制，显著减少重复计费的标记数（Tokens）并降低首个可见字符的响应时间。

### 2.4  Few-shot vs. CoT vs. Caching

| 技术     | 解决的问题               | 核心建议                           |
| -------- | ---------------------- | ---------------------------------- |
| Few-Shot | 输出格式难控制、模式复杂 | 提供 3-5 个“标准答案”作为模版      |
| CoT      | 逻辑推理或计算错误       | 强制要求模型“写出推导过程”再给答案 |
| Caching  | 成本高、响应慢           | 把不动的内容写在最前面             |


## 3 结果导向设计 (Outcome-First Design) 
在 GPT-5 系列模型（如 GPT-5.5 和 GPT-5.4）的开发中，“结果导向”已成为 Prompt 工程的核心范式。与过去需要“手把手”教模型每一步怎么做的指令堆叠不同，现在的最佳实践是**定义终点，而非规定路径**。

### 3.1 核心理念：定义“目的地”而非“每一步路径”
- **理念转变**：GPT-5.5 等现代模型拥有更高效的推理能力，过多的琐碎过程指令（Process-heavy instructions）反而会增加噪声、缩窄模型的搜索空间，甚至导致输出变得机械化。
- **最佳实践**：描述“好的输出是什么样的”（What good looks like），然后赋予模型自主选择搜索、工具调用或推理策略的空间。
- **案例对比**：
  - 过去（**过程导解**）：第一步搜 X，第二步提取 Y，第三步对比 Z，最后写结论。
  - 现在（**结果导向**）：基于提供的上下文和约束，生成一个准确的对比评估结论。成功的标准是结论必须包含引用且逻辑严密。

### 3.2 结果导向提示词的四大支柱
要构建一个高质量的结果导向提示词，源文件建议涵盖以下四个板块：

- **目标结果** (Target Outcome)：明确描述最终交付物的形态和质量标准。
- **成功标准** (Success Criteria)：定义什么样才算“任务圆满完成”。
- **约束条件** (Constraints)：列出那些必须遵守的死规则（如安全规则、必填字段等）。
- **可用上下文** (Available Context)：提供模型完成任务所需的背景信息或数据。

### 3.3 显式停止条件与缺失行为定义
为了防止模型过度生成或在信息不足时胡编乱造，开发者必须设定边界：

- **显式停止条件** (Stopping Conditions)：明确告诉模型在什么情况下应该停止。例如，“当找到 3 个关键证据时停止搜索”或“如果预算超过 1000 Token 则终止”。
- **证据缺失行为** (Missing-evidence behavior)：定义当模型找不到所需事实时的反馈方式。不要让模型自动回答“不”，而是指示其“如果证据不足，请说明缺失的部分，不要推测”。

### 3.4 验证循环 (Verification Loops)
验证循环是结果导向设计中确保准确性的最强工程手段，尤其适用于高风险任务。

- **1> 为什么要进行验证循环**？
  - 即便是最先进的模型也会犯错。验证循环（或称自我检查）是在模型得出初步结论后，指令其在交付最终结果前进行的一步“轻量级自检”。

- **2> 具体实现方式**：
  - **代码验证**：针对编码任务，要求模型编写并运行单元测试，或使用特定的验证命令来检查生成的补丁是否有效。
  - **事实对齐**：针对研究任务，指令模型在返回答案前，检查每一条主张是否在参考资料中有明确的出处。
  - **格式自检**：针对结构化输出（如 JSON），要求模型在完成生成前，对照定义的 Schema 检查是否存在必填项缺失或类型错误。

- **3> 工程案例**：
  - 在执行“不可逆操作”（如修改数据库、扣费）之前，提示词应包含一个特定的“执行框架”：
    ```text
    在返回结果前，请对照 <rules> 板块检查你的响应。如果发现任何违反约束的行为，请重新规划路径并修正。
    ```

### 3.5 避免“绝对化规则”的陷阱

- 灵活决策：除非是安全红线，否则应尽量避免在 Prompt 中大量使用 ALWAYS、NEVER 或 MUST。
- **优先使用决策规则**：对于**需要判断的情况**（如是否调用工具、是否询问用户），应提供“决策规则”而非“死板指令”。这能让模型更好地处理模糊或不确定的情况。


## 4 自动化与迭代 (Automation & Metaprompting) 
在大模型开发进入生产阶段后，手动调整提示词会变得异常低效。本章节的核心在于如何利用模型的能力来优化模型自身的表现，实现开发流程的自动化。

### 4.1 元提示 (Metaprompting)：提示词的自我进化
- **定义**：元提示是指**让 AI 充当自己的“提示词工程师”**。通过向模型展示失败案例和现有的指令，要求模型分析逻辑漏洞并提出改进建议方案。

- **适用场景**：当模型出现以下问题时最为有效
  - **过度思考**：在执行简单任务前浪费大量推理标记。
  - **指令冲突**：系统提示词太长，导致模型顾此失彼。
  - **语气不当**：输出风格变得机械化或过于啰嗦。

- **核心收益**：它避免了开发者盲目猜测哪一行指令导致了错误，而是通过根因分析精准定位并修复提示词。

### 4.2 推理努力 (Reasoning Effort)：智力与成本的自动化平衡
- **参数作用**：这是一个物理级调节旋钮，用于控制模型在回答前投入多少“思考标记（Reasoning Tokens）”。

- **级别划分与匹配**：
  - ``none / low``：适用于执行密集型任务（如字段提取、简单分类），追求极致的速度和低成本。
  - `medium`：GPT-5 系列的默认平衡点，适合大多数编程和通用交互。
  - ``high / xhigh``：仅建议用于复杂的研究、战略写作或多步骤规划。

- 工程实践：建议将其视为 **“最后一英里”调优**。应先优化提示词结构，若性能仍达不到要求，再提升推理努力级别。

### 4.3 高效 Metaprompting 的“两步走”工作流
它将提示词迭代从“凭感觉”变成了“工程化流程”。

#### 第一步：诊断失败 (Diagnose Failures)
不要直接要求模型给出一个新提示词，而是先进行根因分析。

- **操作方法**：在一个独立的 API 调用中，将当前的系统提示词和 **一小批失败的原始日志（Traces）**发送给模型。
- **指令重点**：要求模型充当审计员，找出指令中的模糊之处、相互矛盾的规则，或导致模型偏离目标的诱导性词汇。
- **关键原则**：此时只要求“分析”，不要求“解决方案”，这能防止模型因为急于修改而忽略深层原因。

#### 第二步：打补丁 (Patch the Prompt)
基于第一步的分析结果，进行针对性的指令修复。

- **操作方法**：运行第二个独立的 API 调用，专注于 **“局部微调”而非全盘重写**
- **指令重点**：指派模型生成一个“修复补丁”。例如：“根据上述分析，请重新编写 <instructions> 部分，以消除在处理 A 任务和 B 任务时的逻辑冲突”。
- **循环验证**：完成修复后，必须使用这套新提示词重新运行评估测试（Evals），观察是否修复了旧错误的同时引入了新问题。

#### 指令层级管理 (Instruction Hierarchy)
- **解决冲突**：针对 GPT-5 等高级模型，由于其对指令遵循极度精确，模糊或矛盾的指令比在旧模型中更具破坏性（会浪费大量推理标记去尝试调和矛盾）。
- **优先级定义**：在 Metaprompting 过程中，应显式告诉模型哪些指令是“绝对红线”（如安全规则），哪些是“决策建议”。


## 5 工程化管理 (Engineering Practices)
本章节是 Prompt 工程从“个人技巧”转变为“生产级软件系统”的关键，其核心思想是像管理源代码一样管理提示词。

### 5.1 提示词代码化管理
- **存储建议**：源文件明确建议将生产环境的提示词直接**存储在应用程序代码**中，而不是创建可重用的 API 提示词对象。OpenAI 甚至计划在 2026 年底前逐步停用 API 中的提示词管理功能。

- **模块化**：应将提示词构建逻辑（Prompt Builders）放在靠近其所支持功能的小型模块中。

- **工程优势**：代码化管理允许开发者使用类型化输入（Typed inputs）、进行代码审查（Code review）、编写测试以及使用现有的部署流程来改变模型行为。

### 5.2 提示词模板化 (Prompt Templating)
- **动态插槽**：对于客户数据、文件内容或任务选项等动态值，建议使用类型化函数参数或 ``Schema`` 来处理。
- **输出合同** (Output Contracts)：通过明确定义输出合同（如预定义的 JSON Schema），可以确保模型生成的**结构化数据能被后端程序稳定解析**。

### 5.3 评估与测试 (Evaluation & Testing)
- **基准检查**：在更改生产环境的提示词之前，必须添加代表性的固定装置 (Fixtures)、单元测试和评估检查 (Evaluation checks)。
- **行为监测**：建立测试套件来测量提示词的行为，以便在迭代、更换或升级模型版本时监测性能变化。

### 5.4 工程化部署与迁移流程
确保大模型应用在迭代过程中不发生“性能回退”的核心工程手段。

#### 1> 提示词版本迁移策略
当需要将提示词从旧模型（如 GPT-4）迁移到新模型（如 GPT-5 系列）时，源文件建议遵循 **“单次变动原则”**：

- **切换模型但不改提示词**：首先只更换模型 ID，并固定 ``reasoning_effort``（推理努力）等参数，以测试模型变化本身的影响。
- **建立性能基线**：运行评估套件获取基准分数。
- **针对性调优**：仅在出现性能回归时，才使用提示词优化工具或添加特定的约束（如 Verbosity 控制）来修复性能。
- **按需提升推理努力**：只有在提示词优化无效时，再尝试调高 reasoning_effort 参数。

#### 2> 分阶段发布 (Staged Releases)
- **灰度发布**：利用功能标志 (Feature flags) 或配置系统来滚动提示词的变更，而不是全量覆盖。
- **持续集成**：将提示词变更纳入标准的 CI/CD 流程，确保每次提示词的修改都经过了自动化评估套件的验证。

#### 3> 成本与延迟优化
- **提示词缓存** (Prompt Caching)：在构建消息列表时，将预期会重复使用的内容（如大型上下文、静态规则）放在提示词的最开头。
- **参数前置**：在 JSON 请求体中，也将重复使用的参数排在前面，以最大限度地利用缓存来降低延迟和成本。

### 5.3 角色与权力的等级制度
- **权威定义**：在工程实现中，应明确 developer（开发者消息）拥有比 user（用户消息）更高的优先级。
- **解耦逻辑与数据**：将 developer 消息视为函数定义（业务逻辑），将 user 消息视为函数参数（输入数据）。这种解耦能有效防止提示词注入并提高系统的稳定性。
