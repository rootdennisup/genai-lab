## 1 Chat Completions 与模型快照

### 1.1 Chat Completions（聊天补全）
Chat Completions 是 OpenAI 为 GPT-4 等聊天优化模型提供的 API 端点(LLM 的入口)。与传统的文本补全不同，它要求输入具有明确的结构，以模拟多轮对话的形式。

调用 create 方法时，必须提供以下**两个参数**：

- **model (模型 ID)**：指定要使用的模型，如 gpt-4。
- **messages (消息列表)**：这是一个对象数组，代表了对话的历史记录。每个消息对象必须包含：
  - ``role``：角色，可选值为 system、user 或 assistant。
  - ``content``：该角色的具体对话内容。
  
Chat Completions 示例：
```python
import openai

# 调用 ChatCompletion 端点
response = openai.ChatCompletion.create(
    model="gpt-4",  # 这里可以替换为具体的快照版本，如 gpt-4-0707
    messages=[{"role": "user", "content": "Hello World!"}],
)

# 提取并打印模型生成的响应内容
print(response["choices"]["message"]["content"])
```

### 1.2 模型快照（Model Snapshots）
模型快照是 OpenAI 为了解决大模型在生产环境中的稳定性问题而提供的功能。

- **为什么要锁定版本**？ 
  - 大模型（如 gpt-4）会持续更新。如果开发者直接使用通用名称，模型可能会在一夜之间更新，导致针对相同输入生成的回答发生变化，这对于追求确定性的生产应用来说是巨大的风险。

- **什么是快照？**
  - 快照是模型的静态版本。例如，gpt-4-0613 是 GPT-4 在 2023 年 6 月的一个特定版本。

- **锁定版本的意义：**
  - **行为一致性**：确保模型升级不会导致应用逻辑中断或性能回退。
  - **开发可控性**：开发者可以在准备好后再手动迁移到新版本，而非被动接受更新。


## 2 Messages（消息列表）结构
在 Chat Completions API 中，对话不再是单一的文本字符串，而是一个由消息对象组成的数组。这种结构化的方式让开发者能够精确定义 AI 的身份、历史背景和当前任务。

### 2.1 基本结构
每一个消息对象通常包含两个核心属性：
- **role (角色)**：标识该消息的发送者身份。
- **content (内容)**：具体的消息文本字符串。

### 2.2 核心角色

| 角色           | 定义与作用                                                                                 |
| -------------- | -----------------------------------------------------------------------------------------|
| developer (新) | **开发者角色**。提供应用程序的业务逻辑、系统规则和指令。其权限最高，优先级高于用户消息            |
| system (传统)  | **系统角色**。用于设定助手的行为、语气和身份。在旧版模型中是定义规则的主要方式                    |
| user           | **用户角色**。代表最终用户输入的指令或问题。它像函数的“参数”，被应用于 developer 定义的“函数体”中 |
| assistant      | **助手角色**。由模型生成的消息。存储先前的回复对于保持对话的上下文（记忆）至关重要                |

Messages 结构示例：
```json
messages = [
    # 1. 开发者设定：定义最高优先级的业务规则和助手身份
    {"role": "developer", "content": "你是一个专业的法律助手。只能基于提供的文档回答，严禁虚构事实。"},
    
    # 2. 用户输入：提供具体问题
    {"role": "user", "content": "请解释这份合同的违约条款。"},
    
    # 3. 助手回复：模型基于 developer 规则生成的回答
    {"role": "assistant", "content": "根据合同第三章第五条，违约方需支付总额10%的赔偿金..."},
    
    # 4. 用户追加：进一步追问
    {"role": "user", "content": "那如果是因为不可抗力呢？"}
]
```

### 2.3 developer vs. system 
在 GPT-5 等新一代模型中，developer 角色（或 API 中的 instructions 参数）逐渐取代或增强了 system 角色，其核心区别在于**权限与指令优先级**：

- **指令优先级 (Chain of Command)**:
  - ``developer`` 消息拥有**最高权限**。模型被训练为优先遵循 ``developer`` 提供的指令，即使 user 消息尝试通过“请忽略之前所有指令”等手段（即提示词注入）进行干扰，模型也会更坚定地执行 ``developer`` 设定的规则。
  - 传统的 ``system`` 角色虽然也用于设定行为，但在防范恶意注入时的稳定性不如新一代的 developer 角色。

- **逻辑与输入的解耦**：
  - ``developer`` 和 ``user`` 的关系可类比编程语言中的“函数”与“参数”。
  - ``developer`` 消息定义了**系统的商业逻辑和运行规则（函数定义）**。
  - ``user`` 消息仅作为输入和配置（函数参数），所有的用户输入都会在开发者定义的逻辑框架内被处理。

- **工程建议**：
    - 在开发新应用时，应将核心的身份设定、成功标准、约束条件和工具使用规则放在 developer 消息中。

## 3 Streaming（流式输出）
Streaming（流式输出）是提升大模型应用用户体验最直观、也是最核心的技术手段之一。

### 3.1 Streaming 是如何工作的
在大模型的标准调用模式下，API 会等待模型生成全部回复内容后，才一次性将完整的 JSON 响应返回给开发者。

- **流式模式机制**：当开启流式输出时，模型每预测出一个 Token（标记），API 就会立即将其作为一个“数据块（Chunk）”推送给客户端。
- **交互体验**：这种模式就像你在 ChatGPT 网页端看到的那样，文字是一个接一个“蹦”出来的，而不是在长时间转圈等待后突然跳出一大段文字。

### 3.2 代码中开启流式输出
在调用 ``openai.ChatCompletion.create`` 方法时，只需设置以下可选参数：
- 参数名称：``stream``
- 设置值：将其设为 True（默认为 False）

```python
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "请写一篇关于人工智能的长文。"}],
    stream=True  # 开启流式输出
)

# 此时 response 是一个生成器，需要遍历获取内容
for chunk in response:
    # 提取并打印每个片段的内容
    content = chunk['choices']['delta'].get('content', '')
    print(content, end='', flush=True)
```

### 3.3 对提升“首个可见字符时间”的实际意义
“首个可见字符时间”（Time to First Token, TTFT）是衡量 AI 应用性能的关键指标。开启 Streaming 具有以下重要意义：

- **消除等待焦虑**：模型生成长文本可能需要几十秒甚至数分钟。如果不开启流式输出，用户在这段时间内只能看到加载图标；开启后，用户在不到一秒的时间内就能看到第一个词，从而感知到系统正在积极响应。
- **优化 perceived latency（感官延迟）**：虽然模型生成整段话的总时长没有变，但因为用户可以边看边读，其“感官延迟”被大大降低。
- **配合“前导短语”（Preamble）技术**：针对 GPT-5.5 等新模型，可以在提示词中要求模型先输出一个简短的前导更新（如“我正在为您处理...”），这能进一步缩短用户看到第一个字符的时间，显著改善交互体验。


## 4 Timeout（超时）

### 4.1 什么是 Timeout（超时）
在 API 调用中，超时是指开发者设定的最长等待时间。如果模型在规定时间内没有返回结果，客户端将主动断开连接并抛出异常，防止程序进入无休止的“挂起”状态。

### 4.2 调用LLM 出现超时的原因
LLM 的生成过程具有明显的延迟特征：
- **Token 数量正相关**：响应所需的时间与生成的 Token 数量直接相关。**Token 越多，响应所需的时间就越长**。
- **响应时间上限**：对于复杂的任务或长文本生成，响应过程可能需要几分钟时间。
- **网络波动**：作为远程服务，API 调用受网络环境影响，若不设置超时，单次阻塞可能导致整个后端服务资源耗尽。

### 4.3 超时处理工程实践
处理超时应关注以下策略：
- **合理预估时长**：开发者应根据任务的 max_tokens 预估合理的等待时间。例如，生成简短摘要可能只需 10 秒，而生成数千字的文档可能需要 60 秒以上。
- **开启 Streaming**（流式输出）：这是对抗超时的最佳体验方案。通过流式传输，用户可以在生成过程中持续看到 Token 弹出，从而显著降低感官延迟。
- **使用 Preamble（前导更新）**：针对 GPT-5 等推理模型，在长任务开始前让模型先输出一段短小的确认信息（如“我正在处理...”），可以有效防止客户端因长时间收不到任何数据而误判为超时。
- **代码层设置**：在 Python SDK 中，可以通过参数控制单次请求的超时阈值。

### 4.4 超时后的恢复策略
当发生超时错误时，通常配合重试机制：
- **指数退避**：重试时逐渐增加超时限制，给服务器更多时间处理。
- **断点续传**：针对某些长轨迹任务，可以记录之前的 ID（如 previous_response_id），尝试从之前的推理状态恢复，而不是完全重头开始。


## 5 Retry（重试）
在模型调用的工程实践中，重试不仅是一个技术参数，更是确保 AI 应用从“实验玩具”走向“生产级工具”的防线。

### 5.1 重试对生产环境的重要性
在生产环境中，API 调用不再是百分之百成功的。重试机制的核心意义在于提升系统的**鲁棒性**。

- **应对模型的不确定性**：大模型在设计上是不可靠的，可能会产生幻觉或错误。此外，由于其非确定性，相同的输入在不同时间可能因模型升级或内部波动产生不同的结果。
- **确保任务完备性**：对于长轨迹任务（Long-horizon tasks），模型可能会在未完成所有步骤时提前结束。通过定义**显式的恢复行为（Recovery behavior）**和重试策略，可以强制模型完成复杂的智能体循环（Agent loops）。
- **平滑 API 波动**：API 服务可能会发生变化或出现临时网络问题。将应用与 API 解耦并建立容错机制，能防止 API 的微小波动导致整个业务逻辑中断。

### 5.2 应该针对哪些类型的错误进行重试
并不是所有错误都值得重试。开发者需要根据错误代码的性质采取不同策略：

- **429 错误 (Rate Limit / 限流)**：
  - 原因：由于**每个账户**都有**每分钟请求数（RPM）**和 **Token 数（TPM）**的配额限制，超过限制会触发此错误。
  - 对策：这是重试机制的首要目标。生产最佳实践是采用 **指数退避（Exponential Backoff）**算法，即在重试前逐渐增加等待时间，以避免加剧服务器拥塞。

- **5xx 错误 (Server Errors / 服务器错误)**：
  - 原因：通常由 OpenAI 服务器端的临时波动或超时引起。
  - 对策：这类错误具有**临时性**，通常在几秒钟后的第二次尝试就能成功。

- **逻辑性“软失败”**：
  - 情况：模型虽然返回了 200 状态码，但输出不符合预期的 Schema，或者在验证循环中未通过校验。
  - 对策：针对这类失败，应在代码层触发“逻辑重试”，通过**修复提示词**或增加**推理努力（Reasoning effort）**来重新获取结果。

### 5.3 如何实现重试机制
实现重试不应只是简单的循环，而应包含以下要素：

- **定义完备性契约**（Completeness contract）：在提示词或代码逻辑中明确什么是“任务完成”，若未达成则触发重试。
- **验证循环**（Verification loop）：在执行**高风险动作**（如扣费、修改数据库）之前，增加一个轻量级的**检查步骤**。如果检查失败，则重试生成。
- **持久化中间状态**：对于智能体任务，重试时应传递 previous_response_id，以便模型从之前的推理轨迹中恢复，而不是从头开始，从而节省成本。


## 6 Rate Limit（限流）

### 6.1 Rate Limits 的概念与触发因素
限流是 API 供应商为了保证服务公平分配和系统稳定而设置的保护措施。

触发因素主要包括：
- ``RPM`` (Requests Per Minute，每分钟请求数)：限制你在 60 秒内可以发起的 API 调用总次数。
- ``TPM`` (Tokens Per Minute，每分钟 Token 数)：限制你在 60 秒内发送和接收的 Token 总量。
- **并发限制**：某些情况下还会限制同时进行的活跃请求数。
- **触发后果**：一旦超出上述任一配额，API 将返回 429 错误 (Too Many Requests)。限流通常是**基于账户等级和模型类型**动态调整的。

### 6.2 使用 tiktoken 预防限流错误
建议开发者在调用端点之前使用 OpenAI 提供的 ``tiktoken`` 库。
- **本地预检**：``tiktoken`` 允许你在本地计算文本字符串的标记数，而无需发起网络请求。
- **预防策略**：
  - **输入规模控制**：通过预先计算 Prompt 的 Token 数，确保其不会在单次请求中超出模型上限（如 GPT-4 的 8192 限制），也不会因瞬时发送大量文本而迅速耗尽 TPM 配额。
  - **精确截断**：如果检测到上下文过长，可以利用 tiktoken 进行精确截断，保留最重要的信息同时确保请求合法。
  - **成本估算**：在实际请求前根据 Token 数精确估算每一笔调用的成。

### 6.3 高并发优化建议
针对高并发场景，可以从延迟和吞吐量的方向进行优化：

- **延迟优化** (Latency Optimization)：
  - 开启 ``Streaming``：通过流式输出减少“首个可见字符时间”。
  - **预测输出** (Predicted Outputs)：利用新特性加速特定格式的生成。
  - 调整**推理努力** (Reasoning Effort)：针对 GPT-5 等推理模型，在低延迟需求下可将 ``reasoning_effort`` 设为 none 或 low。

- **吞吐量与成本优化**：
  - **异步调用**：在 Python 中使用 async/await 提高并发处理能力。
  - **批处理**(Batch API)：对于非实时任务，使用 Batch 模式可以获得更高的限流配额和更低的成本（通常为 50% 折扣）。
  - **提示词缓存** (Prompt Caching)：重复使用相同的提示词开头，可以显著降低延迟并减少重复计费的 Token 数。

## 7 推理努力（Reasoning Effort）
推理努力是针对 GPT-5 系列等新一代推理模型引入的一个关键 **API 参数**，它允许开发者精确控制模型在生成最终答案之前的**“思考”程度**。

### 7.1 定义与作用
“推理努力”是一个用于**平衡模型智能、响应延迟和 Token 成本**的调节旋。

- **机制**：模型在回答前会生成内部的“思维链”（Chain of Thought）标记。该参数决定了模型愿意投入多少**推理标记**来分析问题和规划步骤。
- **价值**： 并非所有任务都需要深度思考。通过调节此参数，开发者可以为简单的任务节省成本和时间，为复杂的逻辑推理任务释放模型的全部潜力。

### 7.2 推理努力的五个级别

| 级别      |描述与推荐场景                                                            | 性能特征                 | 
| ---------|-------------------------------------------------------------------------|--------------------------|
|``none`` |最快速度。适用于简单的字段提取、支持分流或短文本转换。完全不使用推理标记，表现类似于 GPT-4o|延迟最低，成本最低|
|``low``  |轻度思考。适用于对延迟敏感但需要处理复杂指令的任务，能比 none 获得小幅的准确率提升| 低延迟                  |
|``medium``|默认平衡点。适合大多数通用交互和交互式编程。 GPT-5 系列的默认设置              | 智能与速度的折中         |
|``high`` | 深度推理。适用于复杂的、多步骤的规划任务或困难的编码挑战                       | 延迟显著增加，成本较高    |
|``xhigh``| 极致智能。仅建议用于长轨迹的智能体任务或需要最高智力水平的科研/策略写作         |延迟最高，最耗标记         |

### 7.3 工程实践：如何选择合适的努力程度
建议开发者将“推理努力”视为 **“最后一英里”的调优手段**，而非首选方案:
- **Prompt 优先原则**：在增加推理努力之前，应先通过增强提示词（如建立完备性契约、添加验证循环、明确工具使用规则）来尝试提升性能。
- **按任务属性分流**：
  - **执行密集型**任务（如工作流步骤、短结构化转换）：建议从 none 开始。
  - **研究密集型**任务（如多文档评估、冲突解决、策略写作）：建议从 medium 或更高开始。
- **提升智能体积极性**：如果发现模型过于频繁地向用户求助或过早终止任务，增加 ``reasoning_effort`` 可以提高模型自主解决问题的毅力（Persistence）。

### 7.4 模型迁移建议（从旧模型升级）
当你将应用从 GPT-4 迁移到具有推理努力参数的模型（如 GPT-5.2）时，请遵循以下路径：
- **固定版本**：首先切换模型，并锁定 reasoning_effort 以匹配旧模型的 profile（例如从 GPT-4o 迁移时先设为 none）。
- **基准评估**：运行评估（Evals）以建立性能基线。
- **按需提升**：仅在评估出现性能倒退且优化 Prompt 无效时，再尝试将推理努力提升一级。
