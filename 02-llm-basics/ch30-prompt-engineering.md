## 1 结构化指令的四大支柱 
为了让 ``developer``（开发者角色）的消息发挥最大效力，建议将其拆分为以下四个逻辑板块。

- **身份 (Identity)**：定义“你是谁”
  - 身份设定是提示词的基石，描述了助手的目的、沟通风格和高层目标。

  - **作用**：设定模型的“心理边界”和语气，引导其进入特定的知识领域。
  - **示例**： “你是一位专业的法律助理，语气严谨且专业” 或 “你是一位经验丰富的软件架构师，擅长发现代码中的安全漏洞”。
  - **核心逻辑**：这种设定可以影响模型在处理模糊信息时的默认决策偏好。

- **指令 (Instructions)**：定义“该怎么做”
  - 提示词中最核心的板块，提供具体的操作指南和行为准则。

  - **内容要求**：规定模型**应该做什么**（任务动作）以及绝对**不能做什么**（安全规则、行为红线）。
  - **子规范**：包含如何调用自定义函数、处理未知情况的原则、甚至输出长度的限制（如“不准超过 50 个字”）。
  - **进化范式**： 在 GPT-5 系列中，指令正从“说明过程”向 **“定义结果”**转变（即下文提到的结果导向设计）。

- **示例(Examples)**：定义“学着做”
  - 即**少样本学习** (Few-Shot Learning)，通过提供 1-5 个具体的“输入-输出”样本，帮助模型识别复杂的模式。

  - **优势**：当单纯的文字指令难以描述复杂的输出格式或转换逻辑时（例如：将文本转换为特定的 Emoji 组合），示例能让模型隐式地“拾取”模式。
  - **编写原则**：示例应当**清晰、相关**，展示多样化的可能输入及其**期望输出**，避免模棱两可。

- **上下文(Context)**：定义“依据什么做”
  - 提供模型回答问题所需的额外信息或背景资料。

  - **构成**：包含私有文档、参考资料、历史对话或特定任务的相关背景（如 RAG 检索回来的知识片段）。
  - **工程建议**：这部分通常建议放在提示词的**末尾**，以确保模型在处理完规则后再读取庞大的参考数据。


## 2 格式化技巧：Markdown 与 XML 标签化
这是工业级提示词的核心技巧，用于帮助模型理解指令的逻辑边界，防止数据与指令混淆，并传达复杂的层级关系。

### 2.1 Markdown：构建层级与可读性
Markdown 是结构化提示词的基石，主要用于**定义文档的整体框架**。

- **标题（Headers）**：使用 #、##、### 等标记来区分提示词的主次部分（如身份设定、操作指令、任务背景等）。这能帮助模型快速识别提示词的层级结构。
- **列表（Lists）**：使用有序或无序列表来列出具体的规则（Rules）或约束（Constraints），使指令条理化，提高模型的指令遵循率。
- **代码块与反引号**：使用反引号（`）或代码块标记文件路径、函数名或代码片段，确保这些技术细节在文本中“脱颖而出”。
- **优势**：Markdown 是 OpenAI 推荐的入门格式，它能显著提高提示词在开发过程中的可读性，且现代模型（如 GPT-4.1 及更高版本）对 Markdown 结构的理解非常稳健。

### 2.2 XML 标签化：精确界定边界与元数据
XML 标签（如 ``<rules>...</rules>``）是工业级提示词中处理复杂数据的利器，尤其在**长上下文任务**中表现优异。

- **界定起止（Delineation）**：XML 标签能精确地告诉模型**某个部分的开始和结束**。例如，将参考资料包裹在 <context> 标签中，能有效防止模型将参考数据误认为是指令本身（即防止数据与指令混淆）。
- **支持嵌套（Nesting）**：XML 允许在不同部分之间进行嵌套。例如，在 <examples> 标签内部，可以为每个示例使用子标签来区分输入和输出。
- **元数据属性（Attributes）**：可以在标签中添加属性来提供额外的元数据。例如：<doc id="1" title="Source">...</doc>。这种方式在**处理多个文档或长文本检索（RAG）**时，能帮助模型准确引用和定位信息。
- **性能表现**：在 OpenAI 的测试中，XML 格式在处理大量文档或长上下文时的**表现远优于 JSON** 格式（JSON 在输入上下文中往往表现较差且需要转义字符）。

### 2.3 工程实践
在实际工程实践中，通常建议将两者结合使用：

- **框架用 Markdown**： 使用 Markdown 标题定义大的逻辑板块（如 # Instructions）。
- **细节用 XML**： 在大的板块内部，使用 XML 标签包裹动态变化的输入数据、多个参考文档或少样本示例（Few-Shot）。

另外，建议：

- **避免在输入端过度使用 JSON**：除非是在编写代码或定义函数参数，否则 JSON 的冗余度和转义需求可能会降低模型在输入端的理解效率。
- **模型特异性**：对于 GPT-4.1 等模型，官方明确指出已经针对 XML 中的信息遵循能力进行了优化，建议优先使用 XML 标签包裹关键背景资料。

## 3 结果导向设计(Outcome-First)
结果导向 标志着提示词设计从“教模型如何做（过程）”向 **“告诉模型交付什么（结果）”**的重大范式转移。

### 3.1 核心理念：定义“目的地”而非“路径”
- **范式转变**：传统的提示词往往倾向于过度说明过程（Process-heavy），即机械地规定第一步、第二步该做什么。
- **为何转变**：GPT-5 等现代模型拥有更高效的**推理能力和自主规划空间**。过多的琐碎过程指令反而会：
  - 增加干扰噪声，缩窄模型的搜索空间。
  - 导致输出变得机械化，或产生不必要的“复述指令”等废话（Noise）。

- **最佳实践**：你应该描述 **“好的输出是什么样的”**（What good looks like），然后赋予模型自主选择搜索、工具调用或推理策略的空间。

### 3.2 结果导向提示词的“四大支柱”
要构建一个高质量的结果导向提示词，通常需要涵盖以下四个逻辑板块：

- **目标结果**(Target Outcome)：精确、具体的描述最终交付物的形态和质量标准（例如：一份包含三个对比维度的结构化财务评估结论）。
- **成功标准**(Success Criteria)：定义什么样才算“任务圆满完成”，作为模型自检和推理的依据。
- **约束条件**(Constraints)：列出必须遵守的硬性规则，如安全红线、必填字段、特定的 JSON Schema 约束等。
- **可用上下文**(Available Context)：提供模型完成任务所需的所有背景资料、证据来源或私有文档。

### 3.3 关键工程组件
在结果导向设计中，为了确保模型在自主规划路径时不偏离目标，需要配合以下工程手段：

- **显式停止条件**(Stopping Conditions)：明确告诉模型何时该终止任务。例如：“当找到 3 个关键证据时停止搜索”。
- **缺失行为定义**(Missing-evidence behavior)：定义模型在证据不足时的反馈方式，严禁推测。例如：“如果找不到事实，请在相应字段填入 null，禁止主观臆断”。
- **验证循环**(Verification Loops)：这是确保准确性的最强手段。指令模型在交付最终结果前进行轻量级自检，例如：
  - **代码验证**：要求模型编写并运行单元测试以检查补丁是否有效。
  - **格式自检**：在返回 JSON 前，对照 Schema 检查必填项是否缺失

### 3.4 案例对比：过程导向 vs. 结果导向 

| 维度      | 传统：过程导向                                   | 现代：结果导向                                   | 
| ---------|-------------------------------------------------|-------------------------------------------------|
| 指令逻辑 | 第一步搜 X，第二步提取 Y，第三步对比 Z，最后得出结论 | 基于提供的资料和约束，生成一份逻辑严密的对比评估报告 |
| 规则倾向 | 大量使用 ALWAYS、NEVER、MUST 等死板指令            | 优先使用“决策规则”，定义何时该搜索、何时该询问      |
| 模型表现 | 容易在执行繁琐步骤时迷失目标或表现机械              | 能够根据任务 shape 自主优化工具调用和推理策略       |

### 3.5 工程实践
如果你正在将应用从旧模型迁移至 GPT-5 系列，建议遵循 **“单次变动原则”**。

- **精简指令**：避免直接搬运旧的冗长提示词，尝试删减琐碎的过程步骤，将其重写为以“产出物质量”为核心的输出合同。
- **配合参数**：结果导向设计配合 ``reasoning_effort`` 参数（推理努力）效果最佳。对于简单的结果提取可设为 none，对于需要深度推理的结果则调高等级。

## 4 思维链(CoT)
思维链 (Chain of Thought, ``CoT``) 是通过鼓励大模型展示**中间推理步骤**，从而模仿人类解决复杂问题过程的提示词技术。

### 4.1 核心逻辑：给模型一张“草稿纸”
大模型生成回答的本质是基于概率的**迭代式文本补全**。

- **痛点**：对于复杂问题（如多步数学题），如果直接要求答案，模型可能在没有充分“规划”的情况下直接生成第一个错误的标记，导致后续整段逻辑崩溃。
- **机制**：CoT 通过在 Prompt 中诱导模型写出中间推导过程，相当于为模型提供了“草稿纸”。模型在输出最终答案前，先通过预测中间推理标记（Reasoning Tokens）来理清逻辑，从而规避计算或逻辑陷阱。

### 4.2 实现思维链的两大路径

#### 1> 零样本思维链 (Zero-shot CoT)
这是最简单、最常用的一种方式。

- **操作方法**：在提示词末尾添加一句 **“让我们逐步思考” (Let's think step by step)**。
- **效果**：这句指令能触发模型自动将复杂任务拆解为多个子步骤进行推理。研究证明，这能显著提升模型在算术推理和逻辑推导上的准确性。

#### 2> 少样本思维链 (Few-shot CoT)
在提示词中包含少量（通常 1-5 个）包含 **“推理过程+最终答案”**的示例。

- **操作方法**：提供一个输入，并在其输出示例中展示详细的拆解逻辑，引导模型模仿这种推理模式。
- **优势**：当零样本 CoT 无法精确控制推理路径或任务逻辑极度复杂时，少样本 CoT 效果更佳。

### 4.3 适用场景与局限性
- **适用场景**：多步算术运算（如复杂乘法）、符号推理、策略性规划问题以及其他逻辑复杂的任务。
- **局限性**：对于模型回答常识性问题或简单的字段提取任务，思维链通常没有显著的性能提升。

### 4.4 GPT-5 演进：从 Prompt 到参数
在 GPT-5 系列（如 GPT-5.5）等新一代推理模型中，思维链已从单纯的提示词技巧演变为原生底层能力。

- **推理标记**(Reasoning Tokens)：这类模型在回答前会自动生成内部思维链。虽然这些标记在最终 API 响应中通常不可见（除非通过特定接口），但它们会被计费。
- **推理努力**(Reasoning Effort)：开发者现在可以通过调节该参数（none 到 xhigh）来物理级控制模型在内部思维链上投入的资源量。
- **成本与延迟权衡**：使用思维链会增加输出 Token 的数量，从而带来更高的成本和更长的响应延迟。

### 4.5 工程实践
- **优先优化 Prompt**：在调高 ``reasoning_effort`` 参数前，先尝试在 developer 消息中通过 XML 标签（如 <verification_loop>）或 CoT 指令来增强模型性能。
- **避免过度使用**：对于简单的文本转换或分类，应将推理努力设为 none，以追求最低延迟和成本。
- **结合验证循环**：在执行高风险动作前，要求模型不仅要展示 CoT 推理过程，还要进行一次“自检”以识别格式或逻辑漂移。


## 5 少样本学习(Few-Shot Learning)
通过在提示词中包含少量（通常为 1-5 个）“输入/输出”示例，引导模型识别模式并完成新任务，而无需重新训练模型权重。

### 5.1 核心机制：模式“拾取”
少样本学习并非修改模型内部的参数（权重），而是利用模型已有的泛化能力。

- **隐式学习**：模型通过示例隐式地**“拾取”**其中的逻辑、格式或语气，并将其应用到当前的输入中。
- **示例驱动**：当文字指令（如“请翻译成法语”）难以描述极其复杂的转换逻辑或特定格式要求时，示例能提供最直观的指导。
- **编写原则**：示例应当清晰且相关，展示多样化的可能输入及其期望输出，避免提供模棱两可的信息导致错误结果。

### 5.2 对比参照
- **零样本 (Zero-shot) vs. 少样本 (Few-shot)**
  - **零样本**：不提供示例，直接描述任务。适用于简单、模型背景知识充足的任务。
  - **少样本**：提供示例。当零样本无法精确控制输出格式或任务逻辑极其复杂时（如将文本转换为特定表情符号组合），少样本学习非常有效。

- **与微调的区别**：少样本学习不修改模型的内部参数，只是利用模型已有的泛化能力；而微调是在特定数据集上重新训练，修改内部权重。


## 6 元提示(Metaprompting)
元提示是指让 AI 充当自己的“提示词工程师”，通过**分析自身的失败案例和现有指令**来优化提示词的**自我进化**技术。

### 6.1 定义与价值
- 元提示将提示词迭代从“凭感觉”转变为一种工程化流程。开发者向模型展示其不符合预期的输出，并要求模型对自身的指令（Instructions）进行审计，提出改进方案。
- 通过这种方式，模型可以自主发现隐藏的歧义、逻辑冲突或诱导性词汇。

### 6.2 适用场景
元提示在解决以下模型行为问题时最为有效：

- **过度思考**(Overthinking)：模型在执行简单任务前浪费了大量推理标记（Reasoning Tokens），导致响应延迟过高。
- **指令冲突**(Instruction Conflicts)：当系统提示词过长时，模型可能会顾此失彼，或在尝试调和矛盾指令时浪费资源。
- **语气与风格不当**：输出风格变得机械化、过于啰嗦，或者前导说明（Preamble）显得生硬且重复。
- **系统能力扩展**：当需要为智能体系统增加新工具或拓宽业务边界时，利用元提示生成新的指令块比手动编写更具一致性。

### 6.3 工业级元提示“两步走”工作流
为了确保效果，建议将元提示过程拆分为两个独立的 API 调用：

- **第一步：诊断失败** (Diagnose Failures)
  - **操作方法**：在一个独立的调用中，将当前的系统提示词和一小批失败的原始日志（Traces）发送给模型。
  - **关键原则**：此时**只要求根因分析，不要求解决方案**。模型应充当审计员，找出指令中模糊或相互矛盾的规则。
  - **注意事项**：反馈应按失败模式逻辑分组（如：将“过度调用工具”与“语气不当”分开分析），否则模型难以理清复杂的错误线索。

- **第二步：打补丁**(Patch the Prompt)
  - **操作方法**：基于第一步的诊断结果，指派模型生成一个针对性的指令“修复补丁”。
  - **关键原则**：专注于局部微调而非全盘重写，以保持提示词的稳定性。
  - **循环验证**：修复后必须重新运行评估套件（Evals），观察是否在修复旧错误的同时引入了性能回退。

### 6.4 工程实践
- **多样本参考**：在进行元提示时，建议多次生成响应并关注各版本之间的共同元素，以过滤掉过于针对特定案例的无效建议。
- **GPT-5 的优势**：GPT-5 系列模型由于对指令遵循极度精确，因此能以更敏锐的洞察力识别出提示词库中的微小矛盾。
- **模板化应用**：开发者可以建立专门的元提示模板，直接询问模型“为了实现目标行为 X 或防止非目标行为 Y，目前的提示词中哪些元素需要添加或删除？”


## 7 提示词缓存(Prompt Caching)
提示词缓存是一种通过复用已处理过的文本内容，旨在显著降低 API 调用成本并优化响应延迟的生产级优化技术。

### 7.1 核心机制：前缀匹配
提示词缓存的工作原理主要基于**前缀（Prefix）匹配**。当开发者向 API 发送请求时，系统会检查提示词的开头部分是否与之前处理过的请求完全一致。如果匹配成功，API 供应商会直接从缓存中读取这部分 Token 的处理结果，而无需再次进行高耗能的推理计算。

### 7.2 核心原则与排列建议
为了最大化缓存的命中率，开发者在构建消息列表时必须遵循 **“静态内容前置”** 的原则：

- **内容排序**：应将预期会重复使用的内容放在提示词的**最开头**。

- 建议存放的**内容类型**：
  - **身份设定**(Identity)：助手的角色定义和语气要求。
  - **核心指令**(Instructions)：业务规则、安全红线和工具调用原则。
  - **大型上下文** (Context)：例如长篇法律条文、公司背景资料或固定的知识库文档。
  - **少样本示例** (Few-shot)：用于引导模型输出格式的静态案例。

- **JSON 参数顺序**：在 Chat Completions 或 Responses API 的 JSON 请求体中，确保重复使用的参数在 JSON 请求体中也排在最前面，以最大化命中率。

### 7.3 提示词缓存的三大收益
在生产环境中使用缓存技术可以带来物理级别的性能提升：

- **显著降低延迟**：由于无需对缓存部分的 Token 重新进行预处理和推理，系统可以大幅缩短**“首个可见字符时间”（Time to First Token, TTFT）**，从而优化用户的感官体验。
- **大幅节省成本**：通常 API 供应商会对缓存命中的 Token 提供大幅度的计费折扣，对于长上下文或高频次调用的应用，这能有效降低总体支出。
- **提升吞吐效率**：缓存减轻了模型服务器的计算负担，有助于在单位时间内处理更多的并发请求。

### 7.4 Few-shot vs. CoT vs. Caching

| 技术                  | 解决的问题               | 核心建议                           |
| --------------------- | ----------------------  | ----------------------------------|
| Few-Shot（少样本学习） | 输出格式难控制、模式复杂  | 提供 3-5 个“标准答案”作为模版        |
| CoT（思维链）          | 逻辑推理或计算错误       | 强制要求模型“写出推导过程”再给答案    |
| Caching（Prompt 缓存） | 成本高、响应慢          | 把不动的内容写在最前面               |

### 7.5 工程实践
- **本地预检**：建议配合使用 tiktoken 库在本地精确计算静态内容的 Token 数，以便更好地规划缓存策略并估算成本节省情况。
- **版本管理**：当提示词中的静态规则（如 Instructions）发生变动时，会导致缓存失效并触发重新计算，因此建议在发布提示词更新前进行基准测试（Evals）。
- **适用场景**：提示词缓存特别适用于多轮长对话、大规模文档问答（RAG）固定格式的结构化数据提取任务。


## 8 工程化管理(Engineering Practices)
工程化管理的核心思想是**像管理源代码一样管理提示词**。

### 8.1 提示词代码化管理
- **存储建议**：源文件明确建议将生产环境的提示词直接**存储在应用程序代码**中，而不是创建可重用的 API 提示词对象。OpenAI 甚至计划在 2026 年底前逐步停用 API 中的提示词管理功能。

- **模块化**：应将提示词构建逻辑（Prompt Builders）放在靠近其所支持功能的小型模块中。

- **工程优势**：代码化管理允许开发者使用类型化输入（Typed inputs）、进行代码审查（Code review）、编写测试以及使用现有的部署流程来改变模型行为。

### 8.2 提示词模板化(Prompt Templating)
- **动态插槽**：对于客户数据、文件内容或任务选项等动态值，建议使用类型化函数参数或 ``Schema`` 来处理。
- **输出合同** (Output Contracts)：通过明确定义输出合同（如预定义的 JSON Schema），可以确保模型生成的**结构化数据能被后端程序稳定解析**。

### 8.3 评估与测试(Evaluation & Testing)
- **基准检查**：在更改生产环境的提示词之前，必须添加代表性的固定装置 (Fixtures)、单元测试和评估检查 (Evaluation checks)。
- **行为监测**：建立测试套件来测量提示词的行为，以便在迭代、更换或升级模型版本时监测性能变化。

### 8.4 工程化部署与迁移流程
确保大模型应用在迭代过程中不发生“性能回退”的核心工程手段。

#### 1> 提示词版本迁移策略
当需要将提示词从旧模型（如 GPT-4）迁移到新模型（如 GPT-5 系列）时，源文件建议遵循 **“单次变动原则”**：

- **切换模型但不改提示词**：首先只更换模型 ID，并固定 ``reasoning_effort``（推理努力）等参数，以测试模型变化本身的影响。
- **建立性能基线**：运行评估套件获取基准分数。
- **针对性调优**：仅在出现性能回归时，才使用提示词优化工具或添加特定的约束（如 Verbosity 控制）来修复性能。
- **按需提升推理努力**：只有在提示词优化无效时，再尝试调高 reasoning_effort 参数。

#### 2> 分阶段发布(Staged Releases)
- **灰度发布**：利用功能标志 (Feature flags) 或配置系统来滚动提示词的变更，而不是全量覆盖。
- **持续集成**：将提示词变更纳入标准的 CI/CD 流程，确保每次提示词的修改都经过了自动化评估套件的验证。

#### 3> 成本与延迟优化
- **提示词缓存** (Prompt Caching)：在构建消息列表时，将预期会重复使用的内容（如大型上下文、静态规则）放在提示词的最开头。
- **参数前置**：在 JSON 请求体中，也将重复使用的参数排在前面，以最大限度地利用缓存来降低延迟和成本。
