## 1 JSON 模式
通过设置 API 参数，开发者可以从物理层面强制模型输出合法的 JSON 字符串。

- ``response_format`` 参数：在调用 Chat Completions API 时，通过设置 ``response_format={"type": "json_object"}`` 开启 JSON 模式 (JSON Mode)。
- **物理级保障**：开启此模式后，模型生成的输出被强制约束为有效的 JSON 字符串，有效防止因模型混入自然语言解释（如“好的，这是你要的 JSON：”）而导致的解析失败。
- 前置要求：在使用 JSON 模式时，通常必须在 Prompt 中显式包含“JSON”字样，指引模型生成该格式。

## 2 Schema 约束：定义数据的“形状”
仅保证是 JSON 还不够，通常需要数据符合特定的业务逻辑结构。

- **JSON Schema 构造**：通过 JSON Schema 定义字段的 type（类型）、description（描述）以及 required（必填项）
  - **类型描述**：为每个字段提供清晰的 description（如“这是一个 SQL 查询”），能显著提高生成准确率。
  - **必填 vs 可选**：在 Schema 中明确划分 required 字段，有助于模型在信息不足时做出决策。
  - **提取完备性**：在处理长文档（如 PDF）提取时，应显式要求模型保持“提取完备性”，防止其在生成过程中由于 Token 限制而中途“偷懒”。

- **工具调用**(Tool Calling)：这是获取结构化数据最可靠的机制。开发者在 tools 字段中定义函数的参数契约，模型负责提取并填充这些参数。

- 案例：让模型从用户对话中提取查询数据库所需的 SQL 参数。
    ```python
    # 定义工具函数的参数契约（JSON Schema 格式）
    product_search_schema = {
        "type": "object",
        "properties": {
            "sql_query": {
                "type": "string",
                "description": "用于在产品表中搜索的 SQL 查询语句。例如：'SELECT * FROM products WHERE price < 2.00'"
            },
            "limit": {
                "type": "integer",
                "description": "返回结果的最大条数"
            }
        },
        "required": ["sql_query"]  # 强制模型必须生成查询语句
    }

    # 将 Schema 封装进 OpenAI 的 tools 参数中
    tools = [
        {
            "type": "function",
            "function": {
                "name": "find_product",
                "description": "根据 SQL 查询从数据库获取产品列表",
                "parameters": product_search_schema
            }
        }
    ]
    ```


## 3 输出合同(Output Contracts) 
在 GPT-5 系列模型（如 GPT-5.4、GPT-5.5）的开发体系中，输出合同是一种从“过程导向”转向“结果导向”的核心工程范式。它不再仅仅是简单的格式要求，而是对模型必须交付的数据形态、字段质量及完成标准的深度约束。

### 3.1 核心定义与设计理念
- **定义**：输出合同是一套明确的**规范**，规定了模型最终产出物应具备的形状、完整性和逻辑约束。
- **结果导向** (Outcome-First)：这是 GPT-5 时代最重要的进阶认知。传统的提示词倾向于教模型“第一步做什么、第二步做什么”，而输出合同则是定义“终点”而非“路径”。
- **赋予自主性**：GPT-5 系列模型拥有更高效的推理能力。通过输出合同描述“什么是好的输出”，模型可以自主规划最有效的搜索、工具调用或推理路径，从而避免机械化的回答或不必要的噪声。

### 3.2 输出合同的四大支柱
为了构建一个高质量的合同，通常需要包含以下板块：

- **目标结果**(Target Outcome)：精确描述最终交付物的最终形态。
- **成功标准**(Success Criteria)：定义什么样才算“任务圆满完成”，作为模型自检的依据。
- **约束条件**(Constraints)：列出必须遵守的硬性规则（如必填字段、安全红线）。
- **可用上下文**(Available Context)：提供完成该合同所需的所有背景数据或证据来源。

### 3.3 技术实现手段

- 1> **JSON Schema 与结构化约束**
  - **数据契约**：通过预定义的 JSON Schema 精确定义字段的 type、description 和 required 属性。
  - **后端对齐**：明确的合同可以确保模型生成的结构化数据能被后端程序（如 Python/数据库）稳定解析，消除非确定性。

- 2> **完备性契约**(Completeness Contract)
  - **防止过早停止**：在长轨迹任务中，模型可能会在完成部分工作后就停止。输出合同通常包含 ``<completeness_contract>``，强制要求模型在退出前对照合同核对所有子任务的覆盖率。
  - **缺失行为定义**：合同中会明确，若证据不足，应在相应字段填入 null 并说明原因，禁止主观推测。

- 3> **验证循环**(Verification Loops)
  - **交付前检查**：合同要求模型在得出初步结论后，进行一步“轻量级自检”。例如：“在返回 JSON 前，检查是否缺失了必填的 ID 字段”。

### 3.4 引入输出合同的工程收益
- **提升 Token 效率**：通过合同约束冗余度（Verbosity），模型可以更简洁地结构化输出，减少废话，从而降低计费成本。
- **增强推理稳定性**：配合 GPT-5 的 reasoning_effort 参数，明确的合同能让模型在深度推理（high/xhigh）时始终对准最终目标，减少长对话中的“信息迷失”。
- **优化感官延迟** (TTFT)：在流式输出中，合同定义的“前导更新”（Preamble）模式能让用户先看到任务进度确认，改善交互体验。


## 4 提示词技巧：确保“纯净”输出
即使开启了 JSON 模式，优秀的提示词仍是成功的保障：

- **负面提示** (Negative Prompts)：明确告知模型“不要在 JSON 代码块之前或之后添加任何解释文字”或“除了 JSON 以外不要回答任何其他内容”。
- **特定后缀引导**：要求模型的输出必须能被 json.loads 接受，或者将重复的格式指令放在提示词的末尾（重复指引法）。
- **结果导向设计**：定义“什么是好的输出”，让模型自主规划填充 JSON 字段的推理路径，而非手把手教它提取步骤。

## 5 输出校验 & 验证循环
在大规模生产环境中，不能完全信任单次生成的输出。

- **自检指令**：在提示词中增加一个**自检步骤**（例如：“在返回 JSON 前，检查是否缺失了必填的 ID 字段”），利用模型的推理能力进行格式纠偏。

- **验证循环机制**：
  - **得出初步结果**：模型生成 JSON。
  - **轻量级验证**：在交付给最终业务逻辑前，指令模型或代码层针对**高风险字段**（如财务金额、SQL 语法）进行核查。
  - **发现冲突并修正**：如果发现违反约束，模型应重新规划路径并修正输出。

- **失败处理与稳定性**：
  - **重试机制** (Retry)：针对格式错误的“软失败”，应配合 **指数退避**（Exponential Backoff） 算法进行逻辑重试，给服务器和模型恢复的时间。
  - **锁定模型快照**：生产环境建议锁定特定的模型快照版本（如 gpt-4-0613），防止模型更新导致的 **JSON 字段漂移或解析逻辑中断**。
  - **Token 预检**：使用 tiktoken 库预先计算输入规模，确保返回的 JSON 数据不会因为超出上下文窗口而被截断，导致 JSON 结构不完整。
