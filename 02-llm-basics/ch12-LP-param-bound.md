## 1 Temperature & Top P 参数
温度 (Temperature) 与 Top P 是控制模型**输出随机性、多样性和一致性**的两个最核心参数。它们共同构成了调节模型“创造力”与“确定性”的天平。

### 1.1 Temperature(温度)
> 温度参数用于调节模型生成响应时的随机程度。

- **底层原理**：模型在预测下一个标记（Token）时，会为词表中的所有潜在标记分配概率。温度通过缩放概率分布来起作用。

- **参数影响**：
  - **T = 0（高度确定）**：模型会始终选择概率最高的标记，产生高度一致且确定的回答。虽然由于模型机制，OpenAI 不保证绝对的 100% 确定性，但输出已达到近似确定。
  - **T 接近 1（默认/平衡）**：这是 API 的默认设置，模型在保持逻辑连贯的同时，具备一定的多样性。
  - **T 趋近 2（极致创造）**：模型选择低概率标记的机会大幅增加。虽然这能带来更强的创造性，但也极其容易产生幻觉、不连贯或无意义的内容。

### 1.2 Top P (核采样)
Top P (核采样)是通过**候选池的动态筛选**，来控制随机性的可选参数。

- **机制原理**：它不再考虑所有的潜在标记，而是通过筛选累计概率达到阈值 P 的标记集合来控制输出。

- **参数影响**：
  - **P 较低**（如 0.1）：模型只会从累计概率加起来达到前 10% 的极少数候选标记中进行选择。这使得输出变得非常保守和集中。
  - **P 较高**（如 0.9 或 1.0）：模型会考虑更广泛的候选标记（覆盖 90% 或 100% 的概率分布），从而使输出更加丰富多样。

### 1.3 概率缩放 vs. 集合筛选

| 维度     |Temperature (温度)                         | Top P (核采样)                                       | 
| --------|-------------------------------------------|------------------------------------------------------|
| 调节机制 | 概率缩放：通过数学公式缩放所有标记的概率分布  | 集合筛选：通过累计概率阈值动态改变候选标记集合的大小   |
| 取值范围 | 通常在 0 到 2 之间，默认值为 1              | 通常在 0 到 1 之间，默认值为 1                        |
| 高值影响 |结果更多样、具创造性，但也易产生不连贯         | 模型考虑的候选标记越多，输出越丰富                     |
| 低值影响 | 回答越一致、越确定（趋向于贪婪搜索）          | 模型只在极少数最高概率的标记中选择，输出越保守          |

### 1.4 工程实践
在实际开发中，这两个参数的设置直接关系到应用的可用性和可靠性：

- **高精度/结构化任务**（建议 T = 0 或低 P）：
  - 在涉及 数据提取、代码生成、SQL 请求、财务分析或事实核查 等任务时，应将温度设为 0。这能显著降低模型产生幻觉的概率，并确保输出符合预定义的 JSON Schema 等结构约束。

- **创意/发散性任务**（建议 T > 0.7 或高 P）：
  - 在进行 头脑风暴、创意写作、营销广告文案或合成数据集生成 时，建议调高参数。例如，在生成营销活动的合成数据时，设置 T = 1 可以获得更具吸引力和差异化的结果。

- **参数组合原则**：OpenAI 官方强烈建议通常**只更改其中一个参数**，而不要同时修改两个参数。

- **结合“推理努力”**：
  - 针对 GPT-5 等新一代推理模型，在调整随机性的同时，还可以配合 reasoning_effort 参数。对于需要精准逻辑的任务，除了设 T = 0 外，还应选择较高的推理级别以确保证据支撑。


## 2 推理努力(Reasoning Effort)*
推理努力是针对 GPT-5 系列等新一代推理模型引入的关键 API 参数。它允许开发者精确控制模型在**生成最终答案之前**投入多少“思维链”（Chain of Thought）标记进行内部规划和分析。

### 2.1 定义与价值
> 推理努力是一个用于**平衡模型智能、响应延迟和 Token 成本**的物理级调节旋钮。

- **机制**：模型在**回答前**会生成内部推理标记。该参数决定了模型愿意投入多少资源来**拆解问题**。
- **价值**：并非所有任务都需要深度思考。通过调节此参数，开发者可以为简单任务节省成本（不使用推理标记），并为复杂任务释放模型的逻辑潜力。

### 2.2 推理努力的五个级别

| 级别      |描述与推荐场景                                                            | 性能特征                 | 
| ---------|-------------------------------------------------------------------------|--------------------------|
|``none`` |最快速度。适合简单的字段提取、支持分流或短文本转换。完全不使用推理标记，表现类似 GPT-4o|延迟最低，成本最低   |
|``low``  |轻度思考。适合对延迟敏感但指令较为复杂的任务，能比 none 获得小幅准确率提升       | 低延迟                  |
|``medium``|默认平衡点。适合大多数通用交互和交互式编程。 GPT-5 系列的默认设置              | 智能与速度的折中         |
|``high`` | 深度推理。适用于复杂的、多步骤的规划任务或困难的编码挑战                       | 延迟显著增加，成本较高    |
|``xhigh``| 极致智能。仅建议用于长轨迹的智能体任务或需要最高智力水平的科研/策略写作         |延迟最高，最耗标记         |

### 2.3 工程实践：如何选择合适的程度
官方建议将“推理努力”视为 **“最后一英里”**的调优手段，而非首选方案：

- **Prompt 优先原则**：在尝试提高推理努力等级前，应先尝试通过优化提示词（如添加<verification_loop>验证循环、明确工具规则）来提升性能。

- **按任务属性分流**：
  - **执行密集型**任务：如提取、分类、结构化转换，建议从 none 或 low 开始。
  - **研究密集型**任务：如多文档综述、策略编写、矛盾解决，建议从 medium 或更高开始。

- **提升智能体积极性**：如果发现模型过于频繁地向用户求助或过早终止任务，增加 ``reasoning_effort`` 可以提高模型自主解决问题的毅力（Persistence）。





















### 2.4 模型迁移建议（从旧模型升级）
当将应用从 GPT-4 或 GPT-4o 迁移到支持该参数的模型（如 GPT-5.4/5.5）时，应遵循以下路径：

- **锁定等级**：切换模型后，首先将 ``reasoning_effort`` 锁定在 none，以匹配旧模型的低延迟 profile。
- **建立基准**：运行评估套件 (Evals) 以建立性能基线。
- **按需提升**：只有在优化 Prompt 无效，且评估结果出现性能倒退时，才尝试逐级调高推理努力等级。

### 2.5 注意事项
- **成本透明**：推理过程中消耗的“思维标记”是计费的，但在 API 最终响应中通常不会显示具体的思维链内容（除非使用特定的 Responses API 支持）。
- **Responses API 优势**：在 Responses API 中，推理标记可以跨工具调用持久化，利用 previous_response_id 可以避免在每一步都重新构建推理计划，从而提高效率。


## 3 图像精细度控制(Image Detail Control)
针对多模态模型（如 GPT-5.4+），开发者可以通过 `image_detail` 参数精确控制模型对图像的理解深度。

- **三个控制级别**:
  - **low（低精细度**）：模型接收低分辨率版本。优点是响应极快，Token 消耗最少，适用于简单的图像分类或仅需了解大致内容的任务。
  - **high（高精细度）**：标准的高保真理解模式，模型会分块处理图像以获取更多细节。
  - **original（原始精细度）**：针对 GPT-5.4 及未来模型。模型直接处理高分辨率原图。

- **工程应用准则**：
  - 对于 OCR（文字识别）、地理位置定位（Localization）、点击精度任务 或 Computer Use（计算机操作） 等对空间敏感的任务，必须显式指定 `original`，而非依赖默认的 auto。
  - 当速度和成本优先于细节（如判断图片中是否有猫）时，应主动切换至 low。


## 4 AI 幻觉(Hallucination)
AI 幻觉是大模型（LLM）应用开发中一个极具挑战性的现象，指的是 AI 自信地给出**错误回答**，或者提供涉及**虚构信息**的回复。

### 4.1 AI 幻觉的特征
AI 幻觉的核心特征是 **“看似合理但事实错误，一本正经的胡说八道”**。

- **特征**：模型生成的文本逻辑通顺、语气坚定，甚至表现得非常专业，但其内容与现实世界的事实不符，或者包含完全捏造的数据、链接、引用和代码。
- **风险**：由于模型在设计上是“不可靠”的，幻觉可能会误导用户，在医疗、法律或财务等关键决策场景中带来严重后果。

### 4.2 为什么模型会产生幻觉
幻觉的产生源于大语言模型的底层工作原理：

- **本质是文本补全**：LLM 本质上是基于概率的文本补全工具，而非基于逻辑的推理机。它通过迭代预测下一个最可能出现的标记（Token）来生成回答。

- **训练数据的局限**：模型知识完全局限于其训练数据（如 GPT-4 原始知识更新至 2021 年 9 月）。如果某些事实在训练数据中出现频率不高，模型无法通过记忆提取正确答案，就会根据语言规律和概率分布“拼凑”出一个回答。

- **缺乏真实逻辑与感知**：模型并没有真正的数学逻辑或实时感知世界的能力。例如，它回答“2+2=4”是因为在训练文本中见过无数次，而非掌握了算术法则。

### 4.3 常见的幻觉示例
- **数学计算**：在处理简单运算时可能正确，但在面对复杂乘法（如 3695 × 123548）时，模型会自信地给出一个看似接近但错误的数字。

- **视觉/链接误导**：模型有时会声称自己“看到”了某些内容。例如要求 ChatGPT 统计一张图片链接中的斑马数量，它可能会由于无法实时访问链接而捏造一个错误的数字并声称已分析。

- **过度自信**：即使模型犯了错，它对自己的错误输出也往往相当自信，甚至会尝试通过逻辑自圆其说。

### 4.4 降低幻觉的技术手段
开发者可以从提示工程和架构设计两个维度来防御幻觉：

- 1> **提示工程技巧** 
  - **“让我们逐步思考”** (Zero-shot CoT)：在提示词末尾添加这句指令会**触发思维链**，强制模型进行中间推理，将复杂问题拆解为多个步骤，能有效提升算术和逻辑推理的准确性。
  - **给予明确的角色和上下文**：为模型定义一个专家身份（如“你是一位经验丰富的律师”），并提供详尽的背景信息，能限制模型生成的随机性空间。
  - **负面提示** (Negative Prompts)：明确告知模型不准做什么。例如：“如果不知道答案，请直接回答‘不知道’，不要尝试虚构”。
  - **少样本学习** (Few-Shot)：在提示词中包含几个“输入-正确输出”的示例，引导模型模仿正确的逻辑模式。

- 2> **工程与架构手段**
  - **RAG 检索增强**：不指望模型凭记忆回答，而是先从可靠数据库（如向量数据库）检索出相关文本，指派模型 **“基于以上事实回答问题”**。
  - **调用外部工具**(Tools/Plugins)：将模型不擅长的任务（如计算、实时搜索）外包给专业工具（如计算器插件、Wolfram Alpha）处理。
  - **参数控制**：将 Temperature（温度）设置为 0，使模型倾向于选择概率最高的标记，从而产出高度一致且确定的回答，减少随机“发挥”。
  
  - **多层防御与核查**：
    - **验证循环** (Verification Loops)：在执行高风险动作前，增加一步轻量级自检，要求模型检查其输出是否符合初始约束。
    - **意图分析层**：在模型回答前，分析用户输入是否包含诱导幻觉或注⼊的恶意指令。


在开发创意型应用（如写故事）时，幻觉可能被视为“创造力”；但在真相至关重要的问答类应用（如医疗咨询、内部知识库）中，必须通过 RAG 或插件系统来严格限制模型的发挥边界。
