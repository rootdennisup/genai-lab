# 上下文工程（Context Engineering）

## 1 短期上下文与窗口管理

### 1.1 什么是上下文窗口 (Context Window)
> 上下文窗口是模型在单次生成请求中能够处理的最大数据限制，其单位为 Tokens（标记）。

- **物理极限**：模型只能“记住”或考虑处于该窗口范围内的数据。一旦输入（Input）与输出（Output）的总和超过这个限制，模型将无法处理请求或导致响应中断。
- **不同模型的差异**：
  - GPT-4：通常提供 8,192 或 32,768 个标记的版本。
  - GPT-4.1/5 系列：支持高达 1,000,000 (1M) 个标记。

- **性能影响**：较宽的上下文窗口可以处理更长的文档和复杂的对话历史，但随着标记数增加，响应所需的时间也会增长（最长可能需要几分钟）。

### 1.2 如何使用 tiktoken 库来管理上下文
由于 API 不会自动存储先前的对话，开发者必须手动构建消息列表。tiktoken 是 OpenAI 提供的一个专门用于计算字符串中标记数量的库。

- **本地预检**：通过 tiktoken，开发者可以在正式发起网络请求前，在本地计算 Prompt 的 Token 数。
- **预防溢出与失败**：通过预计算，可以确保输入内容加上 max_tokens（预期的输出长度）不会超过模型的上下文窗口上限，从而避免 API 报错。
- **精确成本估算**：API 是根据使用的标记数定价的。使用 tiktoken 能够帮助开发者在调用前端就精准估算出单次请求的费用。
- **管理建议**：在处理超长文本（如整本 PDF 或长对话）时，建议配合 tiktoken 进行分块 (Chunking) 或滚动窗口管理，将最关键的信息保留在窗口内。

### 1.3 Prompt Caching（提示词缓存）的排列原则与意义
提示词缓存是一项通过复用已处理文本来优化成本和性能的技术。

- **排序原则**：在构建消息时，务必将预期会反复使用的静态内容放在提示词的**最开头**。
- **静态内容示例**：包括系统身份设定（Identity）、核心指令规则（Instructions）、大型参考文档（Context）或 Few-shot 示例。
- **JSON 参数前置**：在 API 请求的 JSON 主体参数中，也应将这些重复使用的内容作为首批参数传递。

### 1.4 为什么建议将重复内容放在开头？
- **最大化缓存命中率**：缓存机制通常是基于提示词的“前缀（Prefix）”匹配的。如果重复的内容（如长篇法律条文或公司背景）放在开头，API 供应商可以识别出这部分内容是之前处理过的，从而直接从缓存中读取，无需重新计算。
- **降低延迟 (Latency)**：由于无需对缓存部分的 Token 进行重新推理，系统可以显著缩短“首个可见标记时间（Time to First Token）”。
- **节省成本**：缓存命中部分的 Token 通常会有大幅度的计费折扣，从而降低高频次调用的总体支出。

### 1.5 短期上下文 vs. 短期记忆

#### 概念&联系
- **短期上下文**（Short-term Context）：在工程层面，它指的是当前输入给模型的**所有消息列表**（Messages List），包括系统提示词、历史对话以及当前问题。它是模型的**物理输入载体**。
- **短期记忆**（Short-term Memory）：是一个功能性的概念，指模型在单次会话中 **“记住”并利用**之前交互信息的能力。
- **两者的联系**：上下文是记忆的唯一来源。由于现代大语言模型（如 GPT-4）在 API 调用层面是**无状态（Stateless）**的，它们本身并不会自动“记住”之前的请求。为了让模型表现出拥有“短期记忆”，开发者必须手动将之前的对话历史作为“短期上下文”再次发送给 API。

| 维度     | 短期上下文 (Short-term Context)                              | 短期记忆 (Short-term Memory)                                 |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 本质属性 | 物理数据量。受限于模型的上下文窗口                           | 逻辑状态。是一种让应用表现得连贯的交互特征                   |
| 物理限制 | 以 Tokens（标记） 为单位，有严格的上限                       | 只要在窗口限制内，可以存储多轮对话，但信息过长会导致延迟增加 |
| 持久性   | 瞬时性。API 调用结束后，该上下文即在服务端释放（不考虑缓存） | 依赖于开发者的管理策略（如存储在数据库中并随请求带回）       |
| 衡量工具 | 主要使用 tiktoken 库来精确计算其占用的 Tokens 数量           | 更多通过 LangChain 的 Memory 模块等管理逻辑来维护            |


## 2 上下文压缩 & Compaction
- 是解决大模型“长对话遗忘”和“Token 限制”的核心技术，尤其在 GPT-5 系列模型中，Compaction 已演变为一种系统级的底层能力。
- Compaction 是新一代大模型应用的“长期记忆扩容术”。通过将历史记录转化为高密度的加密状态，它让 AI 能够处理超长轨迹的任务和持久的对话，是构建工业级智能体应用的基础设施。

### 2.1 从“摘要”到“损失感知压缩”
在过去，处理超出窗口限制的内容通常依赖于传统的**文本摘要**（Summarization），即通过一个**中间步骤总结先前的对话**。而 Compaction 是 OpenAI 在 Responses API 中引入的一项原生、高性能的底层实现：

- **技术本质**：Compaction 会对先前的对话状态执行一次 **“损失感知压缩”（Loss-aware compression）**。
- **物理形式**：它将繁杂的历史消息（包括工具调用、用户输入和助手回复）转化为加密、不透明的加密内容项（encrypted_content）。
- **关键区别**：
  - **传统摘要**：属于“弱压缩”，由模型手动生成，容易丢失推理细节，且会产生额外的 Token 消耗。
  - **Compaction**：属于“强压缩”，与模型推理逻辑深度集成，能在极大减少 Token 占用的同时，完整保留任务相关的关键状态。

### 2.2 Compaction 的核心价值
- **延长有效窗口**：允许对话持续数个轮次而不触及物理限制，支持数小时甚至更长时间的连续推理。
- **保持长轨迹连贯性**：在执行多步骤、复杂的智能体（Agent）任务时，Compaction 能够确保模型在长达数千个标记的交互后，依然记得最初的目标和已完成的进度。
- **降低性能衰减**：传统的长上下文随着信息增多，往往会出现“Lost in the middle”的性能下降，而 Compaction 后的 opaque 状态有助于维持模型的推理稳定性。

### 2.3 工作流与实现步骤（基于 Responses API）
Compaction 的标准操作流程如下：

- **正常交互**：开发者像往常一样通过 Responses API 发送包含工具调用、输入和回复的消息列表。
- **触发压缩**：当上下文窗口增长到接近限制或达到某个任务节点时，调用 /responses/compact 端点。
  - 注意：发送到该端点的上下文必须先能装入模型当前的物理窗口内。
- **获取加密状态**：该端点会返回一个 encrypted_content 对象。
- **注入后续请求**：在后续的 API 调用中，开发者将这个加密对象作为历史记录的一部分传入，模型即可基于此“ opaque 状态”继续推理，而无需处理原始的冗长 Token。

### 2.4 工程化实践
在生产环境中使用 Compaction 建议：

- **时机选择**：不要每轮对话都进行压缩。应在完成**重大的里程碑**（例如完成了一个复杂的工具调用阶段）后执行。
- **保持 Prompt 稳定性**：在执行 Compaction 并恢复会话后，系统提示词（Instructions）应保持功能上的一致，以避免模型产生行为漂移。
- **不透明性原则**：加密后的状态项对开发者是不可见的，不要尝试解析或修改加密内容，它仅供模型读取。
- **安全与合规**：Compaction 兼容 ZDR（零数据保留） 组织，确保在长会话中的数据处理符合隐私安全标准。

### 2.5 传统技术的补充：映射-归约 (Map-Reduce)
**映射-归约（Map-Reduce）** 是一种传统的摘要策略。

- **原理**：将超长文档（如长视频转录稿）分成多个块，对每个块分别生成摘要（Map），再将这些子摘要合并成一个总摘要（Reduce）。
- **局限**：这种方式虽然能绕过 Token 上限，但相较于 Compaction，其信息丢失率更高，且在逻辑复杂的推理任务中表现较弱。


## 3 长期记忆
大模型（LLM）本身在 API 调用层面通常是无状态（Stateless），长期记忆（Long-term Memory），使 AI 能够记住用户偏好、适应个人需求并随时间持续学习。实现长期记忆的核心技术路径如下：

### 3.1 智能记忆层方案 (以 Mem0 为代表)
Mem0 提供了一个通用的 AI 智能记忆层，其核心逻辑包括：

- **多层级记忆**：能够跨用户（User）、会话（Session）和智能体（Agent）状态无缝保留信息。
- **自适应学习**：模型能够记住用户的偏好并随着时间的推移不断优化交互。
- **高级算法支持**：
  - **实体链接**：提取、嵌入并跨记忆链接实体，以增强召回效果。
  - **时间推理**（Temporal Reasoning）：支持时间感知检索，能够根据查询区分过去发生的事件、当前状态和未来的计划。
  - **多信号检索**：融合语义、BM25 关键词和实体匹配进行并行评分。

### 3.2 检索增强生成 (RAG)
通过外部知识库为 AI 提供“无限记忆”的常用方法，用于解决模型**无法处理超大规模文档或实时信息**的问题。

- **嵌入与向量存储**：将私有数据（如 PDF、公司文档）通过嵌入模型转换为数值向量，并存储在向量数据库（如 Redis 或 Faiss）中。
- **动态注入**：当用户提问时，系统检索出最相关的文本片段，并将其作为上下文注入提示词，让模型基于这些“事实”生成答案。

### 3.4 模型微调 (Fine-tuning)
微调是另一种将长期记忆“固化”到模型内部的方式：

- **权重调整**：通过使用特定领域的大型数据集重新训练现有模型，调整其内部参数。
- **优势**：微调后的模型能记住特定的写作风格、行业术语或内置指令，从而在每次请求时只需输入更短的提示词。

### 3.5 LangChain 记忆模块
LangChain 框架提供了专门的 Memory 接口，用于在链（Chains）或智能体（Agents）调用之间维持状态：
- 对话持久化：例如 ConversationChain 可以自动将对话的历史记录保留在后续的提示词中，模拟出模型具备记忆的能力。

总结对比：
- RAG 适合处理海量的、需要精确检索的外部知识。
- Compaction 适合在超长对话中保持推理轨迹的连贯性。
- Mem0 类的记忆层 则更侧重于对用户个体偏好的持续学习和跨会话管理。
- 微调 适合将极其专业且固定的领域知识转化为模型的“本能”。


## 4 关键信息抽取与结构化
关键信息抽取与结构化（Information Extraction & Structuring）是将**非结构化文本**（如 PDF、邮件、对话）转化为**机器可解析数据**（如 JSON、SQL）的核心技术。

### 4.1 结构化输出 (Structured Outputs)
结构化输出是实现信息抽取最可靠的方式，它确保模型的响应严格遵守预定义的模式。

### 4.2 工业级抽取最佳实践（针对 PDF 与复杂办公流）
针对 PDF 和 Office 自动化工作流的建议：

- **提取完备性**：
  - **痛点**：在处理长文档时，模型往往会因为 Token 限制或“偷懒”而漏掉部分字段。
  - **对策**：在提示词中显式要求模型保持“提取完备性”，并明确定义如何处理缺失字段（如：若信息不足，不要推测，应在字段中填入 null）。

- **区分必填与可选字段**：在定义 Schema 时，明确划分哪些是必须抽取的（Required），哪些是如果存在则抽取的（Optional），这能显著提高抽取的准确率。

- **多文档/多表格序列化**
  - **稳定 ID (Stable ID)**：处理多个文件或表格时，引导模型为每个结果包含一个稳定的标识符（如文件名、合同标题或原始页码范围）。
  - **独立序列化**：建议针对每个文档单独进行序列化结果输出，防止数据混淆。 

### 4.3 关键提示词技巧
为了获得“纯净”的抽取结果，建议使用以下技巧：
- **负面提示与格式锚定**：
  - 使用指令如“除了 JSON 以外不要回答任何其他内容”。
  - 在提示词末尾加入“输出必须能被 json.loads 接受”以辅助解析。

- **坐标与边界定义**：在进行文档定位（如 OCR 框抽取）时，必须明确定义坐标规范，并添加**漂移检查** (Drift check) 以确保空间信息的准确性。
- **重复指引法**：对于较长的输入，在开头和结尾多次重复格式要求，有助于模型在处理海量信息时保持对输出约束的关注。

### 4.4 验证循环 (Verification Loops)
在高风险的结构化任务中，单次生成是不够的。

- **自检指令**：在最终交付 JSON 之前，指令模型执行一步轻量级自检。例如：“在返回前，对照 <rules> 检查你的响应，确保没有缺失必填的 ID 字段”。
- **格式纠偏**：要求模型仅发出目标格式的内容，并在生成的最后一步进行自查，以识别任何格式漂移。


## 6 对话阶段与中间状态维护
对话阶段与中间状态维护（Phase Handling） 是针对 GPT-5 系列模型（如 GPT-5.5）的一项核心技术，旨在解决长轨迹、多步骤任务中“中间过程”与“最终答案”混淆的问题。

### 6.1 什么是 Phase 参数
在 Responses API 中，``phase`` 是一个专门**用于标识助手消息**（assistant-item）所处阶段的参数。

- **技术背景**：在执行复杂的、涉及多个工具调用或需要生成“前导说明（Preamble）”的长任务时，模型会产生中间状态。如果系统无法区分这些状态，可能会误将一个“正在思考”或“正在准备工具”的中间反馈当成任务已经完成的最终答案。
- **作用**：它允许开发者和模型明确区分哪些内容是**中间过程**（Commentary），哪些内容是**最终交付物**（Final Answer）。

### 6.2 Phase 的取值与含义
phase 字段通常包含以下三种值：

- ``"commentary"`` (评论/前导)：表示该条消息属于**前言、计划描述**或对**即将进行的工具调用的解释**。系统应将其视为“工作状态”而非任务结束。
- ``"final_answer"`` (最终答案)：明确标识该条消息是针对用户请求的最终交付内容。
- ``null``：默认状态或未特定指定的阶段。

### 6.3 为什么维护中间状态至关重要？
- **防止过早停止**（Early Stopping）：在 GPT-5.3 Codex 等模型中，如果未正确处理 phase 参数，模型可能会在完成初步计划或发出一段前导更新后突然停止，导致任务中断。
- **增强 UI/UX 体验**：通过识别 commentary 阶段，应用程序可以向用户展示动态的“进度预览”（如：“我正在查找资料...”、“正在编写补丁...”），提高系统的响应感知度。
- **保持推理连贯性**：正确回传 phase 字段能让模型在后续对话中准确引用先前的推理轨迹，而不会在历史记录重建时产生逻辑断层。

### 6.4 工程实践
在开发智能体或长对话应用时，需遵循以下操作准则：

- **状态回传原则**：如果你的应用程序手动将助手的历史输出项重新回传（Replay）到下一次 API 请求中，必须保留并原样传回原始的 phase 值。
- **优先使用引用 ID**：建议使用 ``previous_response_id`` 参数来维持对话状态。在这种模式下，OpenAI 服务端会自动处理 phase 的持久化，无需开发者手动管理复杂的历史列表。
- **角色限制**：phase 参数 **仅支持助手角色（assistant role）**的消息。绝对不要向用户消息（user messages）中添加该字段。

- **模型要求**：正确实现 ``phase`` 参数是 GPT-5.3 Codex 等模型的强制要求；如果 metadata 丢失，会导致性能显著下降。

### 6.5 Phase Handling 的应用场景

| 场景              | phase 的价值                                                                 |
| ----------------- | ----------------------------------------------------------------------------|
| 工具重负载工作流  | 区分“解释为什么要调用工具”和“工具返回的结果”                                     |
| 长轨迹智能体任务  | 确保模型在执行了数小时或数千个 Token 的交互后，依然知道自己是在“执行中”还是“已交付” |
| 代码生成/补丁任务 | 在给出最终代码补丁前，允许模型先进行一段逻辑说明，而不会引发解析错误                |
