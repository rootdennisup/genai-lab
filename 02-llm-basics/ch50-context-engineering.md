# 模块5：上下文工程（Context Engineering）
本模块解决 AI 如何在超长任务和多用户场景下保持连贯性。


## 1 短期上下文与窗口管理

### 1.1 什么是上下文窗口(Context Window)
> 上下文窗口是模型在单次生成请求中能够处理的最大数据限制，其单位为 Tokens（标记）。

- **物理极限**：模型只能“记住”或考虑处于该窗口范围内的数据。一旦输入（Input）与输出（Output）的总和超过这个限制，模型将无法处理请求或导致响应中断。
- **不同模型的差异**：
  - GPT-4：通常提供 8,192 或 32,768 个标记的版本。
  - GPT-4.1/5 系列：支持高达 1,000,000 (1M) 个标记。

- **性能影响**：较宽的上下文窗口可以处理更长的文档和复杂的对话历史，但随着标记数增加，响应所需的时间也会增长（最长可能需要几分钟）。

### 1.2 如何使用 tiktoken 库来管理上下文
由于 API 不会自动存储先前的对话，开发者必须手动构建消息列表。tiktoken 是 OpenAI 提供的一个专门用于计算字符串中标记数量的库。

- **本地预检**：通过 tiktoken，开发者可以在正式发起网络请求前，在本地计算 Prompt 的 Token 数。
- **预防溢出与失败**：通过预计算，可以确保输入内容加上 max_tokens（预期的输出长度）不会超过模型的上下文窗口上限，从而避免 API 报错。
- **精确成本估算**：API 是根据使用的标记数定价的。使用 tiktoken 能够帮助开发者在调用前端就精准估算出单次请求的费用。
- **管理建议**：在处理超长文本（如整本 PDF 或长对话）时，建议配合 tiktoken 进行分块 (Chunking) 或滚动窗口管理，将最关键的信息保留在窗口内。

### 1.3 短期上下文 vs. 短期记忆
- **短期上下文**（Short-term Context）：在工程层面，它指的是当前输入给模型的**所有消息列表**（Messages List），包括系统提示词、历史对话以及当前问题。它是模型的**物理输入载体**。
- **短期记忆**（Short-term Memory）：是一个功能性的概念，指模型在单次会话中 **“记住”并利用**之前交互信息的能力。
- **两者的联系**：上下文是记忆的唯一来源。由于现代大语言模型（如 GPT-4）在 API 调用层面是**无状态（Stateless）**的，它们本身并不会自动“记住”之前的请求。为了让模型表现出拥有“短期记忆”，开发者必须手动将之前的对话历史作为“短期上下文”再次发送给 API。

| 维度     | 短期上下文                                    | 短期记忆                                                |
| -------- | ---------------------------------------------| ------------------------------------------------------ |
| 本质属性 | 物理数据量。受限于模型的上下文窗口               | 逻辑状态。是一种让应用表现得连贯的交互特征                 |
| 物理限制 | 以 Tokens（标记） 为单位，有严格的上限           | 只要在窗口限制内，可以存储多轮对话，但信息过长会导致延迟增加 |
| 持久性   |瞬时性。API 调用结束后，该上下文即在服务端释放（不考虑缓存）|依赖于开发者的管理策略（如存储在数据库中并随请求带回）|
| 衡量工具 |主要使用 tiktoken 库来精确计算其占用的 Tokens 数量 | 更多通过 LangChain 的 Memory 模块等管理逻辑来维护        |


## 2 上下文压缩(Compaction) 
上下文压缩是解决“长对话遗忘”和“Token 物理限制”的系统级核心技术。它被视为构建工业级、长轨迹智能体（Agent）应用的“长期记忆扩容术”。

### 2.1 技术本质：从“弱摘要”到“强压缩”
在传统开发中，处理超出窗口限制的内容通常依赖于文本摘要 (Summarization)，而 Compaction 提供了更高性能的原生实现：

- **损失感知压缩**(Loss-aware compression)：不同于由模型手动生成的文本摘要，Compaction 是 OpenAI 在 Responses API 中引入的深度集成技术。它对先前的对话状态执行压缩，将繁杂的历史记录（工具调用、用户输入、助手回复）转化为高密度的加密表示。
- **物理形式**：压缩后的内容以一个加密且不透明的项 `encrypted_content` 呈现，开发者不应尝试解析或依赖其内部逻辑，它仅供模型在后续请求中恢复状态。

- **关键区别**：
  - **传统摘要**：属于“弱压缩”，由模型手动生成，容易丢失推理细节，且会产生额外的 Token 消耗。
  - **Compaction**：属于“强压缩”，与模型推理逻辑深度集成，能在极大减少 Token 占用的同时，完整保留任务相关的关键状态。

### 2.2 Compaction 的核心价值
- **延长有效窗口**：允许用户对话持续多个轮次而不触及 128k 或 1M 等物理限制，支持长达数小时或涉及数千个标记的连续推理。
- **支持长轨迹任务**：在执行多步骤、复杂的智能体任务时，确保模型在海量交互后依然记得最初目标和已完成的进度，减少性能衰减。
- **维持推理稳定性**：传统长上下文常面临“Lost in the middle”的问题，而 Compaction 后的不透明状态有助于维持模型的逻辑连贯性。

### 2.3 工作流与实现步骤
基于 Responses API 的标准操作流程如下：

- **正常交互**：开发者像往常一样通过 Responses API 发送包含工具调用、输入和回复的消息列表。

- **触发压缩**：当上下文窗口增长到接近限制或达到某个任务节点时，调用 `/responses/compact` 端点。
  - 注意：发送到该端点的上下文必须先能装入模型当前的物理窗口内。

- **获取加密状态**：该端点会返回一个 `encrypted_content` 对象。

- **注入后续请求**：在后续的 API 调用中，开发者将这个加密对象作为历史记录的一部分传入，模型即可基于此“opaque 状态”继续推理，而无需处理原始的冗长 Token。

### 2.4 工程化实践
在生产环境中使用 Compaction 建议：

- **时机选择**：不要每轮对话都进行压缩。应在完成**重大的里程碑**（例如完成了一个复杂的工具调用阶段）后执行。
- **保持 Prompt 稳定性**：在执行 Compaction 并恢复会话后，系统提示词（Instructions）应保持功能上的一致，以避免模型产生行为漂移。
- **不透明性原则**：加密后的状态项对开发者是不可见的，不要尝试解析或修改加密内容，它仅供模型读取。
- **安全与合规**：Compaction 兼容 ZDR（零数据保留） 组织，确保在长会话中的数据处理符合隐私安全标准。


## 3 映射-归约（Map-Reduce）：传统策略
映射-归约（Map-Reduce）超过大模型上下文窗口物理限制的情况。它的核心逻辑是通过 **“分而治之”** 的思想，将大型摘要任务分解为模型可处理的小块。

### 3.1 工作流程
映射-归约策略通常分为三个标准步骤：

- **分块（Chunking）**：将超长文档拆分为多个较小的块，确保每个块的大小都在模型当前的上下文窗口范围内。
- **映射阶段（Map）**：针对每一个独立的小块，分别调用大模型生成该块的子摘要。
- **归约阶段（Reduce）**：将所有生成的子摘要汇总在一起。如果汇总后的文本依然过长，可以再次进行映射；如果长度适中，则由模型对其进行最后的总结，生成总摘要。

### 3.2 技术优势与局限性
- 优势：能够有效绕过 Token 上限，理论上可以处理任意长度的文本。它是早期处理长文本（如 YouTube 视频长文稿）的标准方案。
- 局限：
  - **信息丢失率高**：每一层映射都会丢失细节，最终的摘要可能遗漏跨块的关键信息。
  - **逻辑推理较弱**：在处理逻辑极其复杂的任务（如跨章节的多跳推理）时表现不如原生的大窗口模型或现代压缩技术。
  - **Token 消耗**：虽然解决了窗口限制，但多次调用模型会产生额外的 Token 成本。

### 3.3 工程实践
在实际开发中，开发者可以通过以下方式应用或改进此策略：

- **框架支持**：LangChain 框架提供了专门的“映射-归约链”（Map-Reduce chains），可以自动完成上述的分块、汇总和总结过程。
- 现代对比： 与 `Compaction` 技术相比，映射-归约属于“弱摘要”。Compaction 是系统级的“损失感知压缩”，能以加密状态保留更多推理细节和任务进度，而映射-归约仅产生纯文本摘要。
- **适用场景建议**：如果任务仅涉及简单的内容概述（如新闻监测、趋势观察），映射-归约依然是一个实用且低门槛的选择。但对于需要极高连贯性的智能体任务，建议优先考虑 Compaction 或 RAG（检索增强生成）架构。


## 4 阶段处理(Phase Handling)
阶段处理旨在解决长轨迹、多步骤任务中“中间过程”与“最终答案”混淆的问题。

### 4.1 核心定义：`phase` 参数
在 Responses API 中，`phase` 是一个专门用于标识 **助手消息（assistant-item）** 所处阶段的元数据参数。

- **技术背景**：模型在执行涉及多个工具调用或需要生成“前导说明（Preamble）”的复杂任务时，会产生大量中间反馈。
- **基本功能**：它允许开发者和模型明确区分哪些内容是任务执行中的中间过程，哪些是最终交付物。

### 4.2 phase 的取值与含义
该参数通常包含以下三种核心取值：

- ``"commentary"`` (评论/前导)：表示该条消息属于**前言、计划描述**或对**即将进行的工具调用的解释**。系统应将其视为“工作状态”而非任务结束。
- ``"final_answer"`` (最终答案)：明确标识该条消息是针对用户请求的最终交付内容。
- ``null``：默认状态或未特定指定的阶段。

### 4.3 为什么维护中间状态至关重要？
- **防止过早停止**（Early Stopping）：在 GPT-5.3 Codex 等模型中，如果未正确处理 phase 参数，模型可能会在完成初步计划或发出一段前导更新后突然停止，导致任务中断。

- **增强 UI/UX 体验**：通过识别 `commentary` 阶段，应用程序可以向用户展示动态的“进度预览”（如：“我正在查找资料...”、“正在编写补丁...”），提高系统的响应感知度。

- **保持推理连贯性**：正确回传 phase 字段能让模型在后续对话中准确引用先前的推理轨迹，而不会在历史记录重建时产生逻辑断层。

### 4.4 工程实践
在开发智能体或长对话应用时，需遵循以下操作准则：

- **状态回传原则**：如果你的应用程序手动将助手的历史输出项重新回传（Replay）到下一次 API 请求中，必须保留并原样传回原始的 phase 值。
- **优先使用引用 ID**：建议使用 ``previous_response_id`` 参数来维持对话状态。在这种模式下，OpenAI 服务端会自动处理 phase 的持久化，无需开发者手动管理复杂的历史列表。
- **角色限制**：phase 参数 **仅支持助手角色（assistant role）**的消息。绝对不要向用户消息（user messages）中添加该字段。

- **模型要求**：正确实现 ``phase`` 参数是 GPT-5.3 Codex 等模型的强制要求；如果 metadata 丢失，会导致性能显著下降。

### 4.5 Phase Handling 的应用场景

| 场景             | phase 的价值                                                                 |
| ---------------- | ---------------------------------------------------------------------------- |
| 工具重负载工作流  | 区分“解释为什么要调用工具”和“工具返回的结果”                                     |
| 长轨迹智能体任务  | 确保模型在执行了数小时或数千个 Token 的交互后，依然知道自己是在“执行中”还是“已交付” |
| 代码生成/补丁任务 | 在给出最终代码补丁前，允许模型先进行一段逻辑说明，而不会引发解析错误                |


## 5 长期记忆架构方案
大模型本身在 API 调用层面通常是无状态（Stateless），长期记忆（Long-term Memory）使 AI 能够记住用户偏好、适应个人需求并随时间持续学习。实现长期记忆的核心技术路径如下。

### 5.1 检索增强生成 (RAG)
通过外部知识库为 AI 提供“无限记忆”的常用方法，用于解决模型**无法处理超大规模文档或实时信息**的问题。

- **嵌入与向量存储**：将私有数据（如 PDF、公司文档）通过嵌入模型转换为数值向量，并存储在向量数据库（如 Redis 或 Faiss）中。
- **动态注入**：当用户提问时，系统检索出最相关的文本片段，并将其作为上下文注入提示词，让模型基于这些“事实”生成答案。

### 5.2 智能记忆层 (以 Mem0 为代表)
Mem0 提供了一个通用的 AI 智能记忆层，其核心逻辑包括：

- **多层级记忆**：能够跨用户（User）、会话（Session）和智能体（Agent）状态无缝保留信息。
- **自适应学习**：模型能够记住用户的偏好并随着时间的推移不断优化交互。
- **高级算法支持**：
  - **实体链接**：提取、嵌入并跨记忆链接实体，以增强召回效果。
  - **时间推理**（Temporal Reasoning）：支持时间感知检索，能够根据查询区分过去发生的事件、当前状态和未来的计划。
  - **多信号检索**：融合语义、BM25 关键词和实体匹配进行并行评分。

### 5.3 LangChain 记忆模块
LangChain 框架提供了专门的 Memory 接口，用于在链（Chains）或智能体（Agents）调用之间维持状态：

- **对话持久化**：例如 ConversationChain 可以自动将对话的历史记录保留在后续的提示词中，模拟出模型具备记忆的能力。

### 5.4 固化记忆方案：模型微调 (Fine-tuning)
- **权重调整**：微调是通过特定领域的大型数据集重新训练模型，将知识“固化”到模型的权重中。
- 优势：微调后的模型能记住特定的写作风格、行业术语或内置指令，从而在后续请求中只需更短的提示词，降低了对外部上下文的依赖。

### 5.5 方案对比

| 方案类型    | 适用场景           | 核心特征                             |
| ----------- | ------------------ | ------------------------------------ |
| 短期上下文  | 基础多轮对话       | 手动回传历史，受限于窗口上限         |
| Compaction  | 长轨迹智能体任务   | 系统级损失感知压缩，保留任务关键进度 |
| RAG         | 海量外部文档库     | 基于向量检索的动态上下文注入         |
| Mem0 记忆层 | 用户个性化偏好管理 | 跨会话学习，支持时间推理与实体链接   |
| 微调        | 专业领域本能固化   | 修改模型参数，使模型内置特定领域知识 |
