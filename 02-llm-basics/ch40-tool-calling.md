## 1 工具定义与语义命名

### 1.1 工具定义的标准结构
工具通常在 API 的 tools 字段中通过 ``JSON Schema`` 格式进行定义，其标准结构包含以下三个核心要素：

- **name（工具名称）**：模型的唯一标识符，模型根据该名称选择要执行的动作。
- **description（工具描述）**：这是模型判断是否需要该工具的最重要依据。
- **parameters（参数契约）**：定义工具所需的输入参数及其数据类型（如 string、integer）和约束。

### 1.2 语义化命名(Semantic Naming)的核心准则
由于 LLM 本质上是基于语义理解进行决策的，因此工具名和参数名必须具备 **“语义正确性”**。

- **工具命名应指示目的**：开发者应确保工具名称能清晰反映其功能。
  - **错误示例**：命名为 search。这个名称过于模糊，模型可能无法区分它是搜索网页、搜索内部文档还是搜索数据库。
  - **推荐示例**：命名为 semantic_search 或 web_search_google。清晰的语义能显著降低模型决策的歧义。

- **参数命名应体现用途**：参数名称（如 query 而非 q）应让模型一眼看出该字段应填充的内容。

### 1.3 高质量 description 的编写建议
描述字段不仅要告诉模型工具是什么，更要界定 **“何时调用”**。

- **简洁明了**：建议使用 1-2 句核心描述，说明工具的作用及触发场景。
- **示例锚定**：如果工具逻辑复杂，可以在描述中或专门的 # Examples 板块提供具体的调用示例，帮助模型理解参数逻辑。
- **决策规则而非死板指令**：与其强制要求“必须调用”，不如提供“决策规则”（如：“当用户询问天气时，调用此工具”），赋予模型在模糊场景下的灵活性。

### 1.4 JSON Schema 约束与参数抽取
在 parameters 部分，利用 JSON Schema 的深度约束可以提升提取的准确率：

- **语义化描述每个字段**：为每个参数提供清晰的 description（例如：“这是一个符合 YYYY-MM-DD 格式的日期”），能显著提高生成准确率。
- **明确必填项(Required)**：在 Schema 中划分 required 字段，有助于模型在信息不足时做出决策（如向用户进一步提问而非胡编乱造）。
- **提取完备性**：在处理长文档时，应要求模型保持“提取完备性”，对缺失项填入 null 而非主观推测。

参考案例：
```text
product_search_tool = {
    "type": "function",
    "function": {
        "name": "find_product", # 语义明确：查找产品
        "description": "根据 SQL 查询从数据库获取产品列表", # 清晰的功能描述
        "parameters": {
            "type": "object",
            "properties": {
                "sql_query": {
                    "type": "string",
                    "description": "用于在产品表中搜索的有效 SQL 查询语句。例如：'SELECT * FROM products WHERE price < 2.00'" # 包含格式引导的描述
                },
                "limit": {
                    "type": "integer",
                    "description": "返回结果的最大条数"
                }
            },
            "required": ["sql_query"] # 声明必填参数
        }
    }
}
```


## 2 并行工具调用(Parallel Tool Calling)

### 2.1 定义与优势
- **定义**：并行工具调用是指模型识别出用户意图需要**多个独立的操作来**完成，并在**一次回复中输出包含多个工具调用（Tool Calls）的列表**。

- **优势**：
  - **降低感官延迟**：模型不再需要等待第一个工具结果返回后再决定调用第二个，从而显著减少了“墙钟时间”（Wall-clock time）。
  - **提升吞吐量**：适用于扫描代码库、检索多个向量数据库或执行多实体操作（如同时查询三个城市的天气）。 

### 2.2 API 配置与模型表现
- **参数控制**：在 API 请求中，通常通过 ``parallel_tool_calls`` 参数来开启或关闭此功能（默认为 true）。

- **模型差异**：
  - GPT-5 系列：在并行调用的效率和准确性上有了显著提升，尤其是在处理长上下文和多文档工作流时。
  - GPT-4.1：虽然支持并行调用，但在极少数情况下可能会出现错误。如果开发者发现并行逻辑不稳定，可以建议将 parallel_tool_calls 设为 false。

### 2.3 工程实践与策略建议
为了确保并行调用的稳定性，建议在 **结构化指令**(Instructions) 中遵循以下原则：

- 1> **明确并行与顺序的界限**
  - **适用并行**：当任务之间相互独立且对响应速度要求较高时，应在提示词中明确鼓励并行。
  - **适用顺序**：当任务之间存在依赖关系、语义模糊或涉及不可逆操作（如扣费、删除文件）时，应指派模型使用顺序执行（Sequencing），以确保安全性。

- 2> **排序原则与推理稳定性**
  - **对应顺序**：为了维持模型的推理稳定性，建议工具**响应回传的顺序**与模型**发起调用的顺序**保持一致。
  - **引导指令**：可以在 developer 消息中添加示例，展示如何同时调用多个函数。例如：“当你需要对比两个公司的财务报表时，请同时调用 get_financials 工具两次”。 

- 3> **响应截断策略**
  - 当多个工具并行返回大量数据时，极易超出上下文窗口。
  - **工程对策**：对每个工具结果实施截断（如限制在 10k Token），并采用 **“保留首尾、截断中间”**的策略以保持信息在分布内。 


## 3 特定任务工具应用
在 GPT-5 系列模型（特别是 GPT-5.1 及更高版本）的工具调用体系中，针对编程和系统操作场景引入了专门经过后训练（Post-trained）的特定任务工具。这些工具使得模型能够从“代码生成器”进化为具有实际操作能力的“智能代理”。

### 3.1 apply_patch 工具：结构化文件编辑
apply_patch 是为**代码编辑场景**量身定制的工具，用于通过 **结构化差异（Diffs）** 精确地修改代码库。

- **主要功能**：它允许模型执行三种操作 `create_file`（创建文件）、`update_file`（更新文件）和 `delete_file`（删除文件）。
- **效率优势**：与生成整个文件的全量代码相比，apply_patch **仅处理变动部分**，响应速度更快且更节省 Token。它特别适合迭代式的多步代码重构任务。

- **工程实现**：
  - **响应 API 集成**：官方建议使用 Responses API 的原生命名工具实现。
  - **格式要求**：模型被训练为擅长处理特定的 Diff 格式。虽然也支持 SEARCH/REPLACE 等格式，但最佳实践是不使用行号，并提供精确的旧代码块和替换后的新代码块，以清晰的定界符分隔。

- **可靠性提示**：即使操作失败，apply_patch 有时也可能返回 “Done”。因此，必须在指令中强制模型在应用补丁后通过运行单元测试或 Python 命令来验证更改的正确性。

### 3.2 shell 工具：受控的系统交互
shell 工具允许模型通过受控的命令行界面（CLI）**与本地计算机环境进行交互**，形成一个 **“计划-执行-反馈”** 的闭环。

- **交互机制**：
  - **计划与执行**：模型提出一个 `Shell` 命令，集成层（代码端）执行该命令并将 `stdout`（标准输出）、`stderr`（标准错误）以及 `exit-code`（退出状态码）原样返回给模型。
  - **用途**：常用于检查系统状态、运行公用程序（如 ls、grep）、安装依赖或执行自动化脚本。

- **三种调用方式**：
  - ``shell_command``：执行单次命令。经验表明，将命令作为单个字符串传递比作为列表传递性能更优。
  - ``exec_command``：启动一个长寿命的伪终端（PTY），适用于需要流式输出、REPL（读取-求值-打印循环）或交互式会话的情况。
  - ``write_stdin``：向现有的交互式会话（由 exec_command 启动）发送按键或输入。

- **参数约束**：调用时包含 `timeout`（超时时间）和 `max_output_length`（最大输出长度）参数，以防止命令无限期挂起或返回海量数据撑爆上下文窗口。


## 4 推理持久化与状态维护
在 GPT-5 系列模型（如 GPT-5.4、GPT-5.5）及其配套的 Responses API 体系中，推理持久化与状态维护是构建高性能、长轨迹智能体（Agent）应用的关键技术。它们解决了传统调用中模型“记不住思考过程”以及“容易提前收工”的痛点。

### 4.1 推理持久化(Reasoning Persistence)
推理持久化是指模型在多轮工具调用（Tool Calling）循环中，能够**直接引用并复用先前的推理标记**（Reasoning Tokens/CoT），而无需在每一次工具返回结果后重新从头规划。

- **核心机制**：``previous_response_id``
  - 开发者在发起新的 API 请求时，通过回传前一轮助手的 response_id，模型能够物理级地定位到之前的推理状态。

- **工程收益**：
  - 节省成本：由于不需要重新生成之前的思维链（CoT）计划，极大地减少了推理 Token 的消耗。
  - 降低延迟：模型可以“接着想”而不是“重新想”，缩短了响应时间。
  - 提升稳定性：在 Tau-Bench 等复杂评估中，仅通过开启推理持久化，模型的任务成功率就有显著提升（例如从 73.9% 提升至 78.2%）。

### 4.2 长会话中的“压缩”状态管理(Compaction)
对于持续数小时或涉及数千次交互的极长轨迹任务，传统的上下文窗口会迅速耗尽。此时需要引入 **Compaction（上下文压缩）**技术。

- **实现方式**：通过 `/responses/compact` 端点，将繁杂的历史记录（工具调用、思考过程等）转化为一个加密且不透明的 `encrypted_content` 。
- **状态保留**：这种压缩是“损失感知”的，它能在极大减少 Token 占用的同时，完整保留任务相关的**关键进度**（State）。开发者只需在后续请求中传入此加密状态，模型即可无缝继续工作。

### 4.3 实践建议
为了确保推理与状态的连贯性，在开发时应遵循以下准则：

- **状态回传原则**：如果你的应用手动管理对话历史，必须原样回传助手消息中的 phase 值。丢失 metadata 会导致严重的性能退化。
- **优先使用引用 ID**：相比于手动回传所有历史消息，使用 `previous_response_id` 是最简单且性能最优的路径，因为它由服务端自动处理状态持久化。
- **角色限制**：phase 参数仅支持助手角色（assistant role）。绝对不要将其添加到用户或开发者消息中。
- **里程碑后压缩**：不要每一轮都进行上下文压缩（Compaction），建议在完成重大的任务里程碑（如完成了一个复杂的工具调用阶段）后再执行。


## 5 交互透明度与校验
交互透明度与校验是确保智能体（Agent）从“实验玩具”走向“生产级应用”的关键。它主要解决两个核心问题：用户如何信任 AI 的自主操作（UX 透明度），以及如何确保 AI 调用工具的结果是准确且安全的（可靠性校验）。

### 5.1 交互透明度：工具前导说明(Tool Preambles)
当智能体执行长轨迹任务或调用重负载工具时，用户往往会面临长时间的“黑盒”等待。

- **定义与作用**：要求模型在执行重大工具调用前，先生成 1-2 句简短的自然语言说明，同步 **“正在做什么”**及 **“为什么这么做”**。

- **技术实现**(Phase Handling)：
  - 利用 API 中的 `phase` 参数，将这些说明标记为 commentary（中间过程），从而与最终答案（final_answer）区分开来。
  - 提升 TTFT：在流式输出中，前导说明能显著缩短“首个可见字符时间”，降低用户的感官延迟和焦虑。

- **编写原则：**
  - 非重复性：避免机械地复述指令，应侧重于当前的意图和计划。
  - 针对性：仅在重大步骤或涉及外部侧重（如调用搜索、支付、删除）时触发。

### 5.2 可靠性校验：验证循环(Verification Loops)
即便是最先进的模型（如 GPT-5）也会犯错，验证循环是确保工具调用准确性的“最后防线”.

- **核心机制**：在得出初步结论后，指令模型在交付最终结果或执行不可逆操作前，进行一次“轻量级自检”。

- **应用场景与策略**：
  - **高风险操作校验**：在执行扣费、修改数据库或下发订单前，强制模型检查其输出是否符合初始约束（如检查是否存在必填的 ID 字段）。
  - **代码验证**：针对编程任务（如使用 apply_patch 工具），要求模型编写并运行单元测试或使用特定的 Python 命令来检查补丁是否有效。注意：某些工具可能在失败时仍返回 "Done"，因此必须通过外部逻辑进行二次验证。
  - **格式自检**：针对结构化输出（JSON），要求模型在完成生成前，对照 Schema 检查字段完整性，对缺失项填入 null 而非主观推测。

### 5.3 鲁棒性保障：失败重试与恢复
当校验失败或 API 出现波动时，系统需要具备自我修复能力。

- **重试逻辑**(Retry)：针对“软失败”（如输出不符合预期的 JSON Schema），应在代码层触发“逻辑重试”。生产环境最佳实践是采用 指数退避（Exponential Backoff） 算法，避免加剧服务器拥塞。
- **完备性契约**(Completeness Contract)：在提示词中明确“任务圆满完成”的标准。如果模型未达成标准（如覆盖率不足或跳过必备步骤），则视为任务未完成并触发恢复机制。
- **持久化中间状态**：在多轮工具交互中，配合 previous_response_id 回传推理标记（Reasoning Tokens），确保模型在重试或恢复时能够基于先前的“思考轨迹”继续，而非重头开始，从而降低成本和延迟。
