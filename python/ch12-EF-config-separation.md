## 1 环境变量
环境变量 (Environment Variables) 是操作系统提供的一种动态命名值，存储在系统的内存或配置文件中。它们以“键=值”的形式存在（例如 PATH=C:\Python314），供运行中的进程及其子进程访问，用以获取系统路径、配置参数或安全凭证。是实现程序 **“代码与配置分离”** 的关键。

### 1.1 解决了什么问题
- **硬编码带来的安全隐患**：不应该把数据库密码、API 密钥直接写在 .py 代码里。通过环境变量，敏感信息可以安全地存储在代码之外。
- **多环境切换的复杂性**：开发环境（Dev）和生产环境（Prod）需要不同的配置（如数据库地址）。通过环境变量，可以让同一套代码在不修改任何字符的情况下，根据环境自动切换。
- **程序发现与路径引导**：python3 利用环境变量动态查找解释器的路径，增加了脚本的通用性。

### 1.2 系统中的位置
在 Python 工程（如 FastAPI 项目）中，环境变量的加载遵循 **“由外而内、外部优先”** 的原则。使用 pydantic-settings 等标准库进行配置管理时，系统会自动按照以下优先级从高到低读取配置。

- **1>.系统/进程级环境变量**：通过操作系统设置（永久生效）或在当前终端（PowerShell/CMD）中临时定义的变量。
- **2>.项目配置文件**：存放在项目根目录下的 .env 文件，结合 Pydantic-settings 等库自动将其加载为环境变量。
- **3>.代码默认值**：在 Python 类或函数中定义的默认初始值。

为什么是这样的顺序？为了满足 **“代码与配置分离”** 的工程化需求：
- **允许运行时覆盖**：通过将系统环境变量设为最高优先级，开发者可以在不修改项目 .env 文件或源代码的情况下，通过终端命令或容器配置（如 Docker）直接覆盖任何设置。
- **保护敏感信息**：通过 .env 文件，可以将数据库密码、API 密钥等敏感信息存储在代码之外，避免硬编码带来的安全隐患。

### 1.3 怎么使用
- 在 FastAPI 等框架中，使用 pydantic-settings 库来管理环境变量。
- 可以定义一个配置类，它会自动从系统环境变量或项目中的 .env 文件里读取对应的值，并进行类型检查。


## 2 pydantic-settings 加载 .env 案例
pydantic-settings 库是实现 **“代码与配置分离”** 的工业级标准方案。它能自动从 .env 文件读取配置，并利用 Pydantic 的能力进行类型检查

### 1> 环境准备
安装 FastAPI 和 pydantic-settings 扩展库：
```bash
pip install "fastapi[standard]" pydantic-settings
```

### 2> 创建配置文件 (.env)
项目根目录下创建一个名为 .env 的文件，存放敏感信息或环境变量：[.env 示例](../codes/python_base/.env)

### 3> 定义配置类 (config.py)
创建一个专门管理配置的文件。通过继承 BaseSettings，Pydantic 会自动将 .env 中的键值映射到类属性上，并完成类型强制转换（如将字符串 "10" 转换为整数 10）。[config 示例](../codes/python_base/app/ef/config.py)

### 4> 在 FastAPI 中使用 
通过 依赖注入 (Depends) 的方式在路径操作中使用配置，这样便于后续进行测试和解耦。[fastapi 示例](../codes/python_base/app/ef/fast_get_settings.py)


### 核心原理与注意事项
- **加载优先级**：pydantic-settings 遵循严格的优先级：系统环境变量 > .env 文件 > 代码默认值。这意味着如果你在生产环境（如 Docker）中设置了同名的系统变量，它会覆盖 .env 中的值。
- **运行时验证**：如果 .env 中缺少了没有默认值的必需字段（如 database_url），或者类型不匹配（如 max_connections 填了字母），程序启动时会立即抛出 ValidationError 并停止，从而防止带着错误配置上线。
- **性能优化**：使用 functools.lru_cache 可以避免每次请求都重新读取文件和解析环境变量，确保高效运行。
- **安全性**：务必将 .env 文件列入 .gitignore，不要将其提交到代码仓库。敏感信息应在部署环境通过系统变量注入。
