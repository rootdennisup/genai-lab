## 1 FastAPI 基本应用
[book_api](../codes/python_base/app/ws/book_api.py)

### 1.1 声明式路由
- **定义**：FastAPI 将 HTTP 请求的路径 (Path) 和方法 (Method) 视为 API 的入口契约。通过装饰器（如 @app.get("/")），开发者可以清晰地定义哪些 URL 触发哪些处理函数。
- **优势**：这种方式实现了“一次声明，多种功能”。底层由 Starlette 负责路由管理（映射 URL 到函数），而开发者只需关注业务逻辑的声明。

### 1.2 路径参数的自动校验
> 路径参数直接嵌入在 URL 路径中，用于标识特定资源（如案例中的 {book_id}）

**自动校验原理**：

- **类型转换**：即便客户端通过 URL 传入的是字符串 "101"，FastAPI 也会利用 Pydantic 的强制转换能力将其转为 Python 的 int 类型。
- **约束校验**：在案例中，Path(..., gt=0) 声明了 ID 必须大于 0。如果用户访问 /books/-5，系统会由于校验不通过自动返回清晰的 422 Unprocessable Entity 错误信息，而不会进入函数体执行。

### 1.3 查询参数的自动校验
> 位于 URL 问号之后的部分（如 ?q=python），通常用于过滤或可选设置。

**自动校验原理**：

- **可选性判定**：通过声明 q: Optional[str] = None，该参数被标记为可选。如果在请求中省略 q，函数仍能正常运行并使用默认值。
- **字符串校验**：通过 Query(..., min_length=3)，FastAPI 会自动检查查询字符串的长度。若输入 ?q=py，系统将自动报错并告知开发者输入太短。

### 1.4 类型提示驱动的作用
- **更少 Bug**：大量的校验逻辑从业务代码中剥离，交由基于 Rust 编写的高性能 Pydantic 引擎处理。
- **自动文档**：FastAPI 根据这些类型提示自动生成 Swagger UI（访问 /docs）和 ReDoc（访问 /redoc），在文档中即能看到参数的详细约束和类型要求。
- **编辑器支持**：IDE（如 VS Code）能提供精准的自动补全和静态检查，极大地提升了开发效率。

*_注：FastAPI 自动生成文档的性能开销可以忽略不计，因为它遵循 **“启动时生成、路径隔离访问”**的原则。这种设计在保证开发者获得极佳开发体验的同时，确保了生产环境业务逻辑的高性能运行。_

开发者只需要使用标准的 Python 类型声明一次参数，FastAPI 就会自动接管后续的所有逻辑。
- **输入转换**：自动从网络请求（JSON、路径参数、查询参数、Cookies 等）中读取数据并转换为对应的 Python 类型。
- **数据校验**：利用 Pydantic 校验数据合法性。如果数据不符合声明的类型（如应为 int 却传了 str），系统会自动生成清晰的 JSON 错误信息返回给客户端。


## 2 依赖注入系统 (Depends)
依赖注入系统 (Depends)，解耦认证与数据库连接案例：[decoupled_api](../codes/python_base/app/ws/decoupled_api.py)

### 2.1 依赖注入如何实现“解耦”
- 传统开发模式：在每个路由函数内部手动编写 db = open_db() 和 check_user(token)。这会导致代码大量重复，且一旦数据库连接方式变更，需要修改所有路由。
- **Depends 模式**：将这些逻辑封装在独立的函数中。路由函数不再负责“如何创建连接”或“如何校验用户”，而只是**声明它需要这些资源**。这种方式将“怎么操作资源”与“怎么执行业务”彻底解耦。

### 2.2 数据库连接的自动化管理 (yield)
- 如案例中，get_db_session 使用了 yield 关键字。在 FastAPI 中，这被视为一种异步上下文管理模式。
- 预定义清理：FastAPI 保证无论业务逻辑是否报错，finally 块中的代码（即 yield 之后的部分）都会执行。有效防止了在并发场景下常见的资源泄露问题。

### 2.3 认证逻辑的标准化
- 通过 ``Depends(get_current_user)``，安全认证成为了一个可插拔的组件。
- 异常拦截：如果认证失败（如 token 错误），get_current_user 抛出的 HTTPException 会在进入业务函数前就被 FastAPI 捕获，并直接返回响应给客户端，保护了后端业务逻辑不被非法触发。

### 2.4 依赖解析全流程
当请求到达 /secure-data/ 时，FastAPI 会启动以下自动化引擎：
- **子项解析**：首先运行 get_db_session 并拿到 db 对象。
- **安全性处理**：接着运行 get_current_user 进行鉴权。
- **提取与转换**：利用 Pydantic 对注入的参数进行类型校验（如果声明了类型模型）。
- **业务执行**：所有依赖就绪后，才执行真正的函数体逻辑。
- **自动清理**：函数返回响应后，回到 get_db_session 执行 yield 之后的代码关闭连接。


## 3 自动化文档生成
Swagger UI 是通过 类型提示 (Type Hints) 和 Pydantic 模型 协同驱动的。[swagger_demo](../codes/python_base/app/ws/swagger_demo.py)

### 3.1 自动化文档生成的原理
- **声明式驱动**：当使用 async def 或 def 定义路由并标注参数类型（如 item_id: int）时，FastAPI 会自动识别这些信息。
- **模型驱动方案**：案例中的 Item 类继承自 Pydantic 的 BaseModel。FastAPI 利用 Pydantic 生成 JSON Schema，这些模式定义了 API 预期的输入和输出结构
- **实时更新**：如果修改了代码中的字段（例如给 Item 增加一个 description 字段），当刷新浏览器时，文档会自动同步更新，包含新的请求体结构。

### 3.2 如何访问文档
terminal 中执行命令： ``fastapi dev app/ws/swagger_demo.py`` ，在浏览器访问：
- http://127.0.0.1:8000/docs：进入交互式的 Swagger UI。
- http://127.0.0.1:8000/redoc：进入另一种风格的可选文档 ReDoc。

退出，terminal 中快捷键 ctrl+C

### 3.3 交互式调试步骤
Swagger UI 不仅仅是说明书，也是实时调试平台：
- 可视化参数：文档会清晰展示哪些是路径参数（item_id）、哪些是查询参数（q），以及请求体的具体字段要求。
- “Try it out” 功能：点击接口右侧的 “Try it out” 按钮，原本静态的文档会变为可编辑状态。
- 填写并执行：
    - 在输入框中填入测试数据（例如 item_id 填 5）。
    - 点击蓝色的 “Execute” 按钮。
    - UI 会自动通过网络向你的 API 发送真实的 HTTP 请求。

- 查看结果：界面会实时展示返回的 JSON 响应、HTTP 状态码（如 200 OK）以及完整的 Curl 命令行示例。

### 3.4 自动化校验的实时反馈
如果在 Swagger UI 中尝试发送错误的数据（例如在要求 int 的 item_id 位置输入了字符串 abc），FastAPI 会利用 Pydantic 的校验能力立即拦截请求。
- **错误拦截**：UI 会显示详细的 422 Unprocessable Entity 错误信息，指出哪个字段出错及错误原因。
- **零配置调试**：这一切逻辑（从类型转换到文档反馈）都是由一次类型声明自动完成的，无需开发者编写额外的校验代码。


## 4 SSE 与流式传输
在 FastAPI 的工程实践中，流式传输 (Streaming) 和 SSE (Server-Sent Events) 是处理大数据量或实时通信的高效方案。它们允许服务器在数据准备好时分块发送，而不是等待所有数据就绪后一次性返回，从而显著降低内存压力并提升用户体验。

[stream_api](../codes/python_base/app/ws/stream_api.py)

### 4.1 异步生成器与 yield (核心机制)
- **原理**：在 Python 中，包含 yield 的函数被称为生成器。在异步函数中，yield 能够实现“暂停并发送”的逻辑。
- **非阻塞特性**：每当生成器 yield 一个数据块时，FastAPI 的底层引擎 Starlette 会立即将其通过网络发送给客户端，而无需等待整个循环结束。这对于处理超大文件或无限数据流（如 AI 回复生成）至关重要。

### 4.2 StreamingResponse (Web 支撑层)
- 这是 FastAPI 继承自 Starlette 的一种特殊响应类。它接收一个迭代器或生成器，并维持一个长连接，直到流结束。
- 应用场景：除了 SSE，它也常用于流式传输大视频文件、大型 JSON Lines 数据或动态生成的 CSV 报表。

### 4.3 SSE (Server-Sent Events) 标准
> SSE 是一种基于 HTTP 的标准化技术，允许服务器主动向客户端推送数据。
- **格式要求**：SSE 对数据格式有严格要求。每条消息通常以 **data:  前缀开头，并以两个换行符 \n\n 结尾**。在 Demo 中，手动构建了这种格式以确保浏览器能正确识别。
- **媒体类型**：必须设置 media_type="text/event-stream"，这是浏览器 SSE API (EventSource) 能够正常工作的关键。

### 4.4 高并发性能支持
- **ASGI 驱动**：由于 FastAPI 运行在 Uvicorn 这种高性能 ASGI 服务器上，这种长连接流式传输不会阻塞服务器处理其他请求。
- **内存优化**：相比于一次性将所有数据读取到内存再返回，流式传输极大地降低了内存占用，因为它在任何时刻都只处理当前的一小块数据。
