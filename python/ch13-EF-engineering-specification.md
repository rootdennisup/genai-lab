# 工程基础(Engineering Foundation)--工程规范


## 1 PEP 8 命名规范
PEP 8 是 Python 官方提供的代码风格指南，其命名规范旨在通过统一的视觉标准，提升代码的可读性和可维护性。

### 1.1 核心命名规则分类
| 类型                 | 命名风格                     | 示例               | 备注               |
| -------------------- | ---------------------------- | ------------------ | ------------------ |
| 包 (Packages)        | 全小写，不推荐下划线         | app, utils         | 尽量简短           |
| 模块 (Modules)       | 全小写 + 下划线 (snake_case) | user_service.py    | 保持文件名清晰易读 |
| 类 (Classes)         | 大驼峰 (PascalCase)          | UserModel, Task    | 与 Java 的命名相似 |
| 函数 (Functions)     | 全小写 + 下划线 (snake_case) | get_current_user() | 动作描述应清晰     |
| 普通变量 (Variables) | 全小写 + 下划线 (snake_case) | item_id, is_active | 变量名应具有描述性 |
| 常量 (Constants)     | 全大写 + 下划线              | MAX_RETRY_COUNT    | 通常定义在模块顶层 |

### 1.2 特殊方法与参数命名
在面向对象编程（OOP）中，Python 有一些约定俗成的命名习惯：
- ``self``：实例方法的第一个参数，代表当前实例对象。这是 Python “显式优于隐式”哲学的体现。
- ``cls``：类方法的第一个参数，代表类本身。
- **魔法方法** (Magic Methods)：以双下划线开头并结尾，如 ``__init__``（构造函数）或``__call__``。开发者应避免自定义此类命名的函数，除非是为了实现特定的 Python 协议。

### 1.3 下划线与访问控制约定
Python 没有强制的 private 或 protected 关键字，而是通过命名约定来表达访问限制意图。
- **单前导下划线** (``_variable``)：受保护变量（Protected）。这是一种 **“弱标识”**，暗示该变量仅供内部使用，外部虽然可以访问，但不建议直接调用。
- **双前导下划线** (``__variable``)：私有变量（Private）。触发 **名称修饰** (Name Mangling)，使变量在类外部不易被直接访问，常用于避免子类中的名称冲突。
- **单后置下划线** (``class_``)：当变量名与 Python 关键字（如 class, list）冲突时，习惯在末尾加一个下划线。


## 2 项目结构

现代 Python 项目的目录结构遵循 **“模块化、配置分离、易于测试”** 的工程化原则。一个典型的、面向生产环境（特别是大模型应用开发）的项目结构通常采用根包模式 (Root Package Pattern)。

以下是现代 Python 项目的标准目录结构详解：

### 2.1 典型目录结构示例
```text
my_llm_project/
├── .venv/                 # 虚拟环境文件夹（Python 解释器副本）
├── app/                   # 核心代码文件夹（根包）
│   ├── __init__.py        # 使文件夹成为可识别的 Python 包
│   ├── main.py            # 应用入口，实例化 FastAPI
│   ├── api/               # 路由模块，利用 APIRouter 拆分逻辑
│   │   ├── __init__.py
│   │   └── v1_endpoints.py
│   ├── core/              # 核心配置，如 pydantic-settings 加载
│   │   └── config.py
│   └── models/            # Pydantic 数据模型或数据库模型
│       ├── __init__.py
│       └── schemas.py
├── tests/                 # 自动化测试目录
│   ├── __init__.py
│   └── test_main.py       # 使用 pytest 编写的测试脚本
├── .env                   # 环境变量配置文件（不进入版本控制）
├── .gitignore             # Git 忽略文件清单
├── Dockerfile             # 容器化部署指令
├── requirements.txt       # 依赖清单
└── README.md              # 项目说明文档
```

### 2.2 核心组成部分详解

#### 1> 根包模式 (Root Package Pattern) —— app/
- **作用**：将所有业务逻辑放入一个主文件夹（如 app/ 或 src/），避免项目根目录被大量的 .py 文件淹没。
- ``__init__.py``：Python 通过这个文件来识别包结构，使得你可以在包内执行相对导入（如 from . import models），增加代码的可移植性。
- **模块化拆分**：不要把所有代码写在 main.py。随着项目增大，应利用 APIRouter 等工具将逻辑拆分到不同的子模块中。

#### 2> 确定性运行时 —— .venv/
- **环境隔离**：虚拟环境是 Python 解释器的私有副本。它确保了项目依赖项与系统全局环境隔离，解决了不同项目间依赖冲突的问题。
- 位置：习惯上将虚拟环境文件夹命名为 .venv 并放在项目根目录下。

#### 3> 代码与配置分离 —— .env 与 config.py
- 安全性：敏感信息（如 API Key、数据库密码）应存储在 .env 文件中，而不是硬编码在源码里。
- 自动化加载：结合 pydantic-settings 库，可以定义一个配置类，自动从系统环境或 .env 文件中读取配置并进行类型检查。


#### 4> 依赖管理 —— requirements.txt
- 物理位置：库被安装在 .venv 的 site-packages 文件夹内。
- 逻辑位置：根目录下的 requirements.txt 记录了所有依赖的精确版本。是实现“一键复现”运行环境的基础。

#### 5>  质量保证 —— tests/
- 测试驱动：在 tests/ 文件夹下为每个功能模块编写脚本。
- 工具支持：使用 pytest 框架，仅需在终端输入 pytest 即可自动发现并运行所有测试用例。

### 2.3 工程化操作建议
- **路径处理**：在代码中定位文件或目录时，推荐使用 pathlib 模块。它提供面向对象的接口，能自动处理 Windows 和 Linux 之间的路径分隔符差异。
- **绝对导入 vs 相对导入**：在包内部使用显式的相对导入；在启动应用或进行外部调用时，确保 sys.path 包含根目录（通常由 pytest 或 uvicorn 自动处理）。
- 命名规范 (PEP 8)：参考 <PEP 8 命名规范> 章节。

## 4 日志记录 (Logging)
在 Python 工程开发中，日志记录 (Logging) 是追踪程序运行状态、定位线上问题以及监控复杂逻辑执行（如大模型调用链）的核心工具。

### 4.1 核心定位：标准库与“内置电池”
- **标准库支持**：Python 在标准库中内置了 ``logging`` 模块，它被归类为 **“内置电池（Built-in Batteries）”** 的一部分，用于提供标准化的日志处理方案。
- **解决了什么问题**：相比于简单的 print() 语句，logging 可以将信息分类（如调试、警告、错误），并决定这些信息是输出到控制台、写入文件、还是发送到远程服务器。

### 4.2 核心实现模式：装饰器 (Decorator)
在现代 Python 工程中，日志记录最优雅的实现方式是使用 **装饰器**。

- **不侵入业务逻辑**：利用装饰器“包装”函数的特性，可以在不修改原函数代码的前提下，动态扩展日志功能。
- 典型记录内容：
    - **函数调用信息**：记录哪个函数在何时被调用。
    - **参数与返回值**：自动记录输入参数的具体值以及函数执行后的返回结果，方便复现问题。
    - **性能统计**：记录函数执行的耗时，这在大模型 API 调用场景中尤为重要。

### 4.3 持久化存储：与文件 I/O 的结合
日志记录是文件读写模块的一个经典应用场景。

- **持久化保存**：通过将内存中的运行信息保存到磁盘（如 .log 文件），确保程序关闭或崩溃后，开发者依然可以追溯历史记录。
- **简单的离线备份**：对于中小型应用，直接使用文件系统记录日志是最直接且低成本的方案。

### 4.4 pytest 集成
在自动化测试中，pytest 框架提供了专门的**日志管理机制**（How to manage logging），允许开发者在测试运行时捕获并断言日志输出是否符合预期。

#### 要点说明：
- **分层记录**：用 ``logger.info`` 记录常规流程，用 ``logger.error`` 记录异常。
- **异步兼容**：在 FastAPI 中使用 ``async def`` 配合装饰器，确保在高并发下日志记录不会阻塞主线程。
- **日志轮转**：在生产环境中，建议配合 ``RotatingFileHandler`` 使用，防止单个日志文件过大占满磁盘空间。
