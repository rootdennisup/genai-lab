命名空间（Namespace）和作用域（Scope）是管理变量“可见性”和“生命周期”的核心机制。

## 1 命名空间 (Namespace)
命名空间是变量的 **“容器”**，它建立了 **名称到对象的映射**。在 Python 的底层实现中，大多数命名空间是通过 **字典 (dict)** 实现的。

- **核心作用**：解决了“同名冲突”问题。不同命名空间中的变量可以同名而互不干扰，例如函数 A 和函数 B 都可以拥有变量 x。
  
- **三种层级及其生命周期**：
  - **内置**命名空间 (``Built-in Namespace``)：包含内置函数（如 abs()）和内置异常。
    - 它在 Python 解释器启动时创建，解释器退出时销毁。
  
  - **全局**命名空间 (``Global Namespace``)：记录模块级别的变量，包括函数、类和导入的模块。
    - 它在模块定义被读入时创建，通常在解释器退出时销毁。

  - **局部**命名空间 (``Local Namespace``)：记录函数内部的变量（参数和局部定义的变量）。
    - 它在函数调用时创建，在函数返回或抛出未处理异常时销毁。

命名空间的生命周期取决于对象的作用域，如果对象执行完成，则该命名空间的生命周期就结束。


## 2 作用域 (Scope)
- 作用域是 Python 程序的一个**文本区域**，在这个区域内，某个命名空间是可以被“直接访问”的。
- 作用域决定了你的代码在哪个“容器”（命名空间）里找东西

### 查找顺序：LEGB 原则
当你在代码中使用一个变量时，Python 会按照以下顺序在不同的作用域中搜索该名称：
- ``L (Local)``：最内层作用域，包含局部变量。
- ``E (Enclosing)``：任何外层嵌套函数的作用域（常见于闭包中）。
- ``G (Global)``：当前模块的全局作用域。
- ``B (Built-in)``：最外层，包含内置名称。

异常情况：如果在这四个地方都找不到，Python 会抛出 NameError 异常。


## 3 关键字：global 与 nonlocal
当需要在函数内部修改非局部变量时，必须显式声明，否则 Python 会默认在局部作用域创建新变量。

- **global 关键字**：
  - 如果你在函数内部想修改全局变量，必须先用 ``global`` 声明。
  - 它告诉解释器：“去最外层的全局作用域找这个变量”。如果该变量不存在，则会在全局作用域创建它。

- **nonlocal 关键字**：
  - 用于嵌套函数中。如果你想修改外层（但非全局）函数中的变量，需要使用 nonlocal 声明。
  - 它告诉解释器：“去外层的函数里找一个现成的变量”，如果外层函数里找不到，则会报错。

## 4 工程实践中命名空间的意义
- **避免污染**：在 FastAPI 等 Web 框架中，为了保证每个请求处理逻辑的独立性，建议减少全局变量的使用，转而通过“依赖注入”来传递配置。
- **虚拟环境的隔离**：虚拟环境 (venv) 虽然主要隔离物理路径上的库，但在逻辑上确保了不同项目拥有独立的全局命名空间，避免了不同项目间依赖包的版本冲突。
- **类与封装**：每个类对象 (Class Object) 都会创建自己的命名空间，这是实现面向对象封装特性的基础。

## 5 异常处理 (Exception Handling)
异常处理 是确保程序健壮性、防止因非预期错误（如网络波动、非法输入或文件丢失）导致系统崩溃的关键机制。

### 5.1 错误与异常的区别
Python 将程序问题分为两大类：
- **语法错误 (Syntax Errors)**：在代码运行前，解释器解析阶段发现的错误（如缩进不当或括号不匹配），程序无法启动。
- **异常 (Exceptions)**：在代码执行过程中发生的错误，即使语法正确，特定的操作（如除以零、读取不存在的文件）也会引发异常并中断程序。

### 5.2 异常处理基础
目的是“捕获”错误并执行替代方案，而不是直接崩溃。

- **try 块**：包含可能抛出异常的代码。
- **except 块**：当指定类型的异常发生时，执行该代码块。可以捕获特定异常（如 ValueError），也可以使用 Exception 捕获所有常规异常。
- **else 块**：如果 try 块中没有发生任何异常，则执行此块。
- **finally 块**：无论是否发生异常，都会执行的代码块。通常用于释放系统资源（如关闭数据库连接或文件）。

### 5.3 异常处理进阶
- **raise 关键字**：使用 raise 显式地抛出一个异常。例如，在 Pydantic 自定义验证器中，如果数据不合规，可以抛出 ValueError。
- **用户自定义异常**：通过继承 Python 内置的 Exception 类，可以创建符合特定业务逻辑的异常类型。
- **with 语句**：在处理文件读写、网络连接或并发锁时，手动编写 try...finally 释放资源显得冗长且易出错，可以使用 with 语句自动释放资源。

### 5.4 异常处理的工程应用及建议
在 FastAPI 和 Pydantic 等框架中，异常处理被提升到了自动化接口响应的高度：
- **Pydantic 校验异常 (ValidationError)**：当输入数据不符合定义的 BaseModel 形状时，Pydantic 会抛出详尽的错误信息。它不仅说明哪里错了，还会列出错误原因（如“应为整数却传入了字符串”）。
- **FastAPI 错误处理 (HTTPException)**：
    - FastAPI 提供了专门的 HTTPException，用于向客户端返回特定的 HTTP 状态码（如 404 或 401）和错误详情。
    - **自动化响应**：如果 Pydantic 校验失败，FastAPI 会自动拦截 ValidationError 并将其转化为带有 422 状态码的 JSON 错误信息返回给前端，无需开发者手动编写 try...except。
- **网络异常 (urllib.error)**：在调用外部 API 时，URLError 及其子类 HTTPError（如 404 错误）能够帮助开发者精准定位网络请求失败的原因。

异常处理的工程建议：
- **保持上下文简洁**：with 块或 try 块中的代码应只包含与资源或风险逻辑相关的操作。
- **精准捕获**：尽量捕获具体的异常类，避免滥用过于宽泛的 except Exception，以免遮蔽掉真正的系统 Bug。
- **测试先行**：在 pytest 中，可以利用 pytest.raises() 断言代码是否在特定条件下正确抛出了预期的异常
