# 模块1：LLM 基础认知


## 1 Token（标记）
Token（标记）是大模型最基础的“原子”单位，理解 Token 是掌握模型行为、控制开发成本和优化应用性能的第一步。

### 1.1 Token 是如何从文本中拆分出来的
在大模型处理文本之前，必须先通过一个名为 **分词器（Tokenizer）**的工具将原始文本转换为模型能理解的数字序列。

- **拆分单位**： Token 并不等同于单词。它可能是单词、单词的一部分（子词）、空格或标点符号。例如，一个长单词可能会被拆分成多个 Token。
- **分词器差异**： 几乎每个语言模型都配有自己特定的分词器。例如，GPT-4 和 GPT-5 的分词方式可能不同。
- **直观换算**： 对于英文文本，理解 Token 数量的一条经验法则是：**100 个 Token 大约等于 75 个英文单词**。
- **计算近似值**：在某些场景下，也可以通过 字节数 / 4 来粗略估算 Token 数。

### 1.2 为什么说 Token 是 LLM 计费的基础
与传统的云服务（按调用次数或按时长计费）不同，LLM API 是严格按量计费的，这个“量”就是 Token 的数量。

- **双向计费**： API 调用的总成本取决于 **输入文本（Prompt）和输出文本（Completion）** 的 Token 总和。
- **计费透明**： 每次 API 调用返回的响应对象中都会包含一个 ``usage`` 字段，明确列出 ``prompt_tokens``（输入数）、``completion_tokens``（输出数）和 ``total_tokens``（总数）。
- **价格差异**： 不同模型的单价不同。例如，GPT-5 的单价通常远高于 GPT-4。开发者通常需要根据每 1000 个 Token 的单价来预估业务成本。

### 1.3 理解 Token 数量限制对开发者有什么实际意义
在工程实践中，Token 的数量直接决定了应用的可用性、响应速度和经济性。

- **上下文窗口（Context Window）限制**： 模型处理信息的“内存”是有限的。每个模型都有其最大的 Token 容量（如 GPT-4 有 8192 或 32768 Token 的版本）。如果对话的总长度超过这个上限，模型将无法工作或被迫忽略之前的上下文。
- **延迟监控**：Token 数量越多，模型生成响应所需的时间就越长。长文本处理甚至可能需要几分钟时间。
- **成本控制**：开发者可以通过设置 max_tokens 参数来限制模型生成的最大长度，从而防止意外的高额账单。
- **开发工具**：强烈建议开发者使用如 ``tiktoken`` 这样的 Python 库，在调用 API 之前先在本地计算文本的 Token 数，以便进行精确的成本估算和输入控制。


## 2 分词器（Tokenizer）
分词器（Tokenizer）是模型处理文本的第一道工序，也是开发者理解模型行为和成本控制的核心。

### 2.1 基本概念
标记化（Tokenization）是将原始文本（如句子、段落）切分成一个一个名为 Token（标记） 的单元的过程。

- **语义独立性**：分词器旨在保证每个标记拥有相对完整和独立的语义，以便后续任务（如生成嵌入或模型输入）使用。
- **Token 的形式**：标记并不等同于单词。它可能是完整的单词、单词的一部分（子词）、空格或标点符号。

### 2.2 处理流程
当模型接收到一段提示词（Prompt）时，分词器的处理流程如下：

- 原始文本（输入） →
- 分词器处理（标记化过程）：
  - 拆分：按特定算法将文本切分为 Token 序列。
  - 映射：将这些标记转换为模型能理解的数字 ID。
- 后续环节：
  - 嵌入（Embeddings）：将数字 ID 转换为数值向量。
  - 模型预测：根据上下文预测下一个标记。

### 2.2 分词器与后续环节的关系
- 衔接嵌入（Embeddings）：由于模型只能处理数值，分词器切分出的 Token 随后会被转换为数值向量（即嵌入），从而让计算机能够高效处理它们之间的语义关系。
- 迭代补全：在文本补全过程中，模型会根据已有的 Token 上下文，逐个预测并添加下一个标记，直到形成完整的响应。


## 3 上下文窗口（Context Window）

### 3.1 什么是上下文窗口
上下文窗口是大语言模型在生成新信息时可以处理的 **“内存”限制**（物理限制）。它以 Token 为单位，规定了模型在单次预测任务中能够“同时看到”的最大文本范围。

- **物理隐喻**：可以把它想象成模型的“短期记忆”。只有在这个窗口内的信息，模型才能理解其中的语义关系并保持逻辑连贯。
- **无状态特性**：需要注意的是，OpenAI 的 API 是**无状态**的，它不会记住之前的请求。因此，如果想进行多轮对话，必须每次都将之前的对话历史完整地作为上下文发送给模型。

### 3.2 不同模型的窗口容量差异
不同模型拥有不同大小的上下文窗口，这直接决定了它们处理长文档或长对话的能力：

- GPT-4 系列： 提供 8,192 和 32,768（约 2.4 万个单词或 40 页文本）两种版本。
- GPT-4.1 ： 窗口大小已显著提升至 12.8 万（128k） 甚至 100 万（1M） 个标记。

### 3.3 上下文窗口对开发者的意义
理解这个限制对应用开发有三个核心意义：

- **功能天花板**：标记总数（输入 + 输出）必须小于模型上限。如果内容超过窗口，模型将无法正常工作，或者被迫忽略最早的上下文信息，导致“丢三落四”。
- **成本与延迟**：上下文中的 Token 越多，API 的计费越高，且模型生成响应所需的时间也越长（长文本处理可能需要几分钟）。
- **复杂推理**：虽然大窗口能处理更多数据，但在处理极长上下文时，模型的性能可能会在需要精确检索或多跳推理的任务中有所下降。

### 3.4 内容超过窗口限制时的工程策略
当对话内容或文档长度超过模型“内存”时，开发者通常采用以下策略：

- **上下文压缩（Compaction）**：通过损失感知（loss-aware）的压缩 pass，将之前的对话状态转换为加密的、更精简的表示，从而在不命中窗口上限的情况下保留关键任务信息。
- **摘要与映射-归约（Map-Reduce）**：将长文本拆分为小块分别生成摘要，再将摘要汇总后再输入模型。
- **RAG（检索增强生成）**：这种方法不将整本书塞进上下文，而是先通过 **嵌入（Embeddings）** 技术在向量数据库中搜索最相关的片段，只把这几百字的“精华”喂给模型。


## 4 Temperature & Top P 参数
Temperature（温度）和 Top P 是控制大语言模型输出随机性和创造力的核心参数。

### 4.1 Temperature（温度）
- **用途**：该参数用于调节模型生成响应时的随机程度。
- **底层原理**：模型在预测下一个标记（Token）时，会为所有潜在标记分配概率。当 Temperature 设置为 0 时，模型会始终选择概率最高的那个标记，从而产生高度一致且确定的回答。
- **随机性控制**：随着温度值的升高（通常在 0 到 2 之间），模型选择概率较低标记的机会增加，使得输出结果更加多样化和具创造性。
- **默认值**：API 的默认值通常为 1。

### 4.2 Top P (核采样)
- **作用**：Top P 是另一种控制模型输出随机性的可选参数。
- **机制**：与 Temperature 一样，是影响模型输出一致性的关键变量。如果更改该参数，即使提示词相同，也可能得到截然不同的结果。

### 4.3 Temperature vs. Top P

| 维度     |Temperature (温度)                         | Top P (核采样)                                           | 
| --------|-------------------------------------------|----------------------------------------------------------|
| 基本定义 | 用于调节模型生成响应时的随机程度和创造力      | 另一种控制模型输出随机性的可选采样参数                      |
| 调节机制 |概率缩放：通过缩放所有潜在标记的概率分布来起作用|概率核筛选：通过筛选累计概率总和达到 P 阈值的标记集合来控制输出|
| 取值范围 | 通常在 0 到 2 之间，默认值为 1               | 通常在 0 到 1 之间，默认值为 1                            |
| 高值影响 |值越高，结果越多样化、更具创造性，但也更容易产生不连贯的内容|值越高（接近 1），模型考虑的候选标记越多，输出越丰富|
| 低值影响 | 值越低（趋近 0），回答越一致、越确定          | 值越低，模型只在极少数最高概率的标记中选择，输出越保守        |


## 5 模型能力边界
虽然大语言模型（LLM）展现了惊人的智能，但作为开发者，必须清醒地认识到它们并不是万能的“超级大脑”。

### 5.1 模型公认的局限性
- **计算能力有限**：GPT 模型本质上是**文本补全工具**，而非计算器。它通过预测下一个标记来生成回答，这在处理简单运算（如 2+2）时可能因为训练数据中见过多次而正确，但在面对复杂计算（如 3695×123548）时往往会“自信地”给出错误答案。
- **信息时效性（知识截断）**：模型无法直接访问互联网获取新信息，其知识完全**局限于训练数据**。例如，GPT-4 的原始知识更新仅截止到 2021 年 9 月（虽然后续版本有所更新，但仍存在截断点）。
- **AI 幻觉（Hallucination）**：模型可能会虚构事实或提供涉及虚构信息的回答，且表现得非常自信。在大模型设计上它们是不可靠的，可能会误导用户。
- **无法执行真实动作**：纯模型本身只能生成文本，无法直接预订航班、订购食品或修改你的本地文件。

### 5.2 通过插件（Plugins）弥补边界
插件是扩展 GPT-4 功能的革命性工具，它让模型成为一个“智能的 API 调用者”。

- **增强计算与功能**：通过激活计算器插件，模型可以准确执行数学运算。
- **获取实时信息**：插件允许模型检索最新的体育比分、股票价格或新闻。
- **连接外部服务**：插件使模型能够代表用户执行操作，如通过第三方服务预订航班或订餐。
- **访问私有知识**：公司可以开发插件，让模型检索内部文档、个人笔记等基于知识的信息。

### 5.3 通过架构设计弥补边界
开发者还可以通过特定的软件架构来增强模型能力：

- **RAG 架构（检索增强生成）**：
  - 不指望模型直接知道答案，而是先从外部数据库（如 Redis）检索出相关的文本摘录，再要求模型根据这些“事实”生成答案。
  - 这种“信息还原”而非“信息检索”的方法，可以有效解决模型不知道私有数据或新数据的问题。

- **智能体与工具（Agents & Tools）**：
  - **多步推理**：利用 LangChain 等框架构建智能体，它可以决定何时调用维基百科搜索或计算器工具。
  - **自我纠正**：智能体可以采取一系列行动，将工具的输出重新输入到上下文中，直到解决复杂问题。

- **微调（Fine-tuning）**：
  - 通过在特定领域的数据集上重新训练模型权重，可以提高模型在专业任务（如法律文本分析、财务摘要）上的准确性。


## 6 幻觉（Hallucination）

### 6.1 什么是“AI 幻觉”
AI 幻觉是指 AI 自信地给出一个错误的回答，或者涉及虚构信息的现象。

- **特征**：模型生成的文本逻辑通顺、语气坚定，但其内容与现实世界的事实不符，或者包含完全捏造的数据、链接和引用。
- **示例**：
  - 在计算复杂乘法（如 3695×123548）时，模型会给出一个看似合理的错误数字；
  - 或者在被要求统计某张图片中的物体数量时，捏造出一个错误的数字并声称自己已经“看到”了。

### 6.2 为什么模型会产生幻觉
幻觉的产生源于大语言模型的底层工作原理：

- **本质是文本补全**：GPT 模型本质上是基于概率的文本补全工具，而非基于逻辑的推理机。它通过迭代预测下一个最可能出现的标记（Token）来生成回答。
- **训练数据的局限**：如果某些事实或复杂计算在训练数据中出现的频率不高，模型无法通过“记忆”提取正确答案，就会根据语言规律和概率分布“拼凑”出一个回答。
- **缺乏真实逻辑**：模型并没有真正的数学逻辑或实时感知世界的能力。它回答 2+2=4 是因为在训练文本中见过无数次，而非它掌握了算术法则。

### 6.3 降低幻觉的提问技巧（Prompting）
开发者可以通过以下提示工程技巧显著降低幻觉发生的概率：

- **“让我们逐步思考”** (Zero-shot CoT)：在提示词末尾添加这句指令，强制模型进行中间推理，将复杂问题拆解为多个步骤。这被证明能有效提升算术和逻辑推理的准确性。
- **给予明确的角色和上下文**：为模型定义一个专家身份（如“你是一位经验丰富的律师”），并提供详尽的背景信息，能限制模型生成的随机性空间。
- **负面提示（Negative Prompts）**：明确告知模型不准做什么。例如：“如果不知道答案，请直接回答‘不知道’，不要尝试虚构”。
- **提供少量样本（Few-Shot Learning）**：在提示词中包含几个“输入-正确输出”的示例，引导模型模仿正确的逻辑模式。

### 6.4 降低幻觉的工程手段（Engineering）
在应用架构层面，开发者应采取以下硬性手段：

- **RAG（检索增强生成）**：不让模型凭记忆回答，而是先从可靠数据库（如 Redis）检索出相关的文本摘录，指派模型“基于以上事实（FACTS）回答问题”。
- **调用外部工具（Plugins/Tools）**：对于模型不擅长的任务（如计算、实时搜索），通过插件系统（如计算器插件、Wolfram Alpha）将任务外包给专业的工具处理。
- **参数控制**：将 Temperature（温度）设置为 0。这能让模型每次都倾向于选择概率最高的标记，从而产出高度一致且确定的回答，减少随机“发挥”导致的幻觉。
- **多层防御与核查**：
    - **意图分析层**：在模型回答前，分析用户输入是否包含诱导幻觉的恶意指令。
    - **验证循环**：在执行高风险操作前，增加一个轻量级的自我检查步骤，要求模型检查其输出是否符合初始约束。
