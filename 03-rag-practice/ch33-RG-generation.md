# 在线检索与生成阶段 (Retrieval & Generation)--生成模块 (Generation)

Generation 是整个系统的核心输出环节。目标是利用检索到的相关知识，结合大模型（LLM）的推理能力，生成准确、可靠且具有参考依据的回答。

## 1 Prompt 拼接
Prompt 拼接 (Prompt Stitching)是将用户原始问题、检索出的上下文知识以及特定的角色指令，按照预定义的模板组合成一段完整的文本输入给大模型。

- **Prompt 模板** (PromptTemplate)：框架通常预置了各种模板（如 text_qa_template），其中包含变量（如 {context_str} 和 {query_str}），这些变量在运行时会被检索到的内容和用户问题实时填充。
- **指令约束**：为了保证生成质量，Prompt 中通常会包含明确的指令，例如“仅基于提供的上下文回答”、“不要编造”等，以规范大模型的行为。
- **实例化输出**：通过 format 或 format_messages 方法，模板可以被转换为普通字符串或支持多轮对话的消息封装类型。


## 2 上下文构造 
上下文构造(Context Construction)决定了检索出的多个知识块（Nodes）如何呈献给大模型。LlamaIndex 提供了多种 **响应模式（Response Modes）**来处理这一过程。

- **合并策略**(Compact)：这是默认模式，它会尽可能将多个知识块合并成大的文本块，以充分利用大模型的上下文窗口，减少调用次数并降低成本。
- **迭代细化**(Refine)：逐个使用知识块进行生成。它先用第一个块生成初始答案，再将该答案与下一个块结合进行“细化”，直到处理完所有知识块。这种模式适合生成非常详细的答案。
- **树状总结**(Tree Summarize)：对所有知识块进行递归总结，直到生成最终答案，非常适合回答概括性或总结性的问题。
- **上下文增强**：有时为了提高生成质量，会采用“分离检索与生成知识块”的策略。例如使用句子窗口检索，检索时使用单句以确保精确，但生成时会自动替换为该句子前后的完整窗口内容（如前后各3句）作为上下文。


## 3 引用来源
为了提高答案的可信度和透明度，生产级的 RAG 应用通常需要展示溯源引用(Reference Sources)。

- 可追踪引用：系统会记录生成答案所依据的 Node 及其元数据（如文件名、页码、相似度评分）。
- 展示方式：在最终生成的答案后，通常会附带“来源节点”列表，让用户可以查看原始文档片段，从而验证信息的真实性。

## 4 拒答策略
拒答策略(Refusal Strategy)是抑制大模型产生“幻觉”（即一本正经地胡说八道）的重要手段。

- **Prompt 指令拒答**：在 Prompt 中明确告知模型：“如果上下文中不包含足够的回答问题的信息，请回答‘我暂时无法回答该问题’，不要编造”。
- **结构化答案过滤**(Structured Answer Filtering)：在 refine 或 compact 模式中，可以开启过滤功能，让大模型自动排除掉与问题不相关的知识块，从而避免噪声干扰生成的准确性。
- **相关性评估**：在一些高级范式（如 C-RAG）中，会在生成前增加一个评估环节。如果评估器认为检索出的内容均不相关，则会触发拒答或通过网络搜索补充信息。

## 5 输出
- **流式输出**(Streaming)：为了提升用户体验，响应生成器应支持流式输出，让用户在大模型生成第一个 token 时就能立即看到反馈，而无需等待全文生成完毕。
- **结构化输出**：通过 output_cls 参数或输出解析器（Output Parser），可以强制要求大模型输出 JSON 或特定的 Pydantic 对象，方便下游系统处理。
